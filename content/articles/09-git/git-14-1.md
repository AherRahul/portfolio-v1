# Git Objects

At its core, Git is a **content-addressable filesystem** — a simple key-value store. Every piece of data in Git is stored as an "object" and identified by its SHA-1 hash. Understanding Git objects helps you truly understand how Git works under the hood.


## The Four Types of Git Objects

Git has exactly four types of objects:

1. **Blob** — Stores file contents
2. **Tree** — Stores directory structure
3. **Commit** — Stores a snapshot with metadata
4. **Tag** — Stores an annotated tag object


## 1. Blob Objects

A **blob** (Binary Large Object) stores the raw contents of a file. It contains ONLY the file content — no filename, no permissions.

```bash
# Store a string as a blob
echo "Hello, Git!" | git hash-object --stdin
# Output: 8ab686eafeb1f44702738c8b0f24f2567c36da6d

# Store a file as a blob
git hash-object src/app.js
# Output: abc1234...
```

A blob's SHA-1 hash is based entirely on the file content. Two files with identical content have the same hash (Git deduplicates automatically).


## 2. Tree Objects

A **tree** represents a directory. It contains:
- References to blob objects (files)
- References to other tree objects (subdirectories)
- Filenames and permissions

```bash
# View a tree object
git cat-file -p main^{tree}
# Output:
# 100644 blob abc123... README.md
# 040000 tree def456... src
# 100644 blob g9h012... package.json

# Mode 100644 = regular file
# Mode 040000 = directory (tree)
# Mode 100755 = executable file
# Mode 120000 = symbolic link
```


## 3. Commit Objects

A **commit** object represents a specific snapshot of the repository. It contains:
- A reference to the root tree (all files at that point)
- References to parent commits
- Author and committer information
- The commit message

```bash
# View a commit object
git cat-file -p HEAD

# Output:
# tree abc1234... (root directory tree)
# parent def5678... (previous commit - none for first commit)
# author Jane Doe <jane@example.com> 1704067200 +0000
# committer Jane Doe <jane@example.com> 1704067200 +0000
#
# Add user authentication
```

The parent field is what forms the **commit history chain**. Merge commits have two parents.


## 4. Tag Objects

An **annotated tag** object contains:
- A reference to the tagged object (usually a commit)
- Tagger information
- The tag message
- Optional GPG signature

```bash
# View a tag object
git cat-file -p v1.0.0
# Output:
# object abc1234...
# type commit
# tag v1.0.0
# tagger Jane Doe <jane@example.com> 1704067200 +0000
#
# Release version 1.0.0
```


## How Objects Connect: A Complete Picture

When you make a commit, here's the object structure created:

```
Commit Object
  ├── tree → Root Tree Object
  │           ├── blob → README.md contents
  │           ├── blob → package.json contents
  │           └── tree → src/ subdirectory
  │                       ├── blob → app.js contents
  │                       └── blob → auth.js contents
  └── parent → Previous Commit Object
```


## The Object Store

All objects are stored in `.git/objects/`:

```bash
ls .git/objects/
# ab/
# de/
# pack/

# Objects are stored with the first 2 chars of hash as directory name
# and the remaining as filename
ls .git/objects/ab/
# c1234567890abcdef...
```

You can inspect any object with:

```bash
# Show the type of an object
git cat-file -t <hash>   # blob, tree, commit, or tag

# Show the content of an object
git cat-file -p <hash>

# Show summary info
git cat-file -s <hash>   # size in bytes
```


## Why This Matters

Understanding the object model helps you:
- Understand why commits are immutable — the hash covers everything
- Understand how rebasing creates new commits (new hashes)
- Understand why Git is efficient — identical files share the same blob
- Debug complex Git situations using low-level tools

Every operation in Git — committing, branching, merging — is ultimately just reading and writing these four types of objects.
