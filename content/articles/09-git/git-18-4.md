# Code Review Best Practices

Code review is one of the most valuable practices in software development. When done well, it improves code quality, spreads knowledge across the team, and catches bugs before they reach production.

## The Value of Code Review

**For the reviewer:**
- Understand what's changing and why
- Catch bugs and design issues early
- Share domain knowledge

**For the author:**
- Get a second perspective
- Learn from feedback
- Document design decisions

**For the team:**
- Consistent code style
- Shared ownership
- Knowledge transfer

## Before Submitting a PR for Review

As the **author**, set your reviewer up for success:

1. **Keep PRs small** — Aim for < 400 lines changed. Large PRs are hard to review well.

2. **Write a clear description:**
   ```markdown
   ## What does this PR do?
   Adds OAuth2 Google login to the authentication flow.
   
   ## Why?
   Users requested social login to reduce friction (#234).
   
   ## How to test?
   1. Click "Login with Google"
   2. Complete OAuth flow
   3. Should be redirected to dashboard
   
   ## Screenshots
   [Before/After]
   
   ## Notes for reviewer
   The oauth callback URL must be configured in env vars.
   ```

3. **Self-review before submitting** — Read your own diff first.

4. **Add inline comments** for complex logic that needs explanation.

5. **Ensure CI is green** before requesting review.

## Reviewing Code Effectively

As a **reviewer:**

### Look For

```
✅ Correctness — Does the code do what it's supposed to?
✅ Security — Are there XSS, injection, or auth vulnerabilities?
✅ Performance — Any O(n²) loops? Missing indexes? N+1 queries?
✅ Error handling — Are edge cases and errors handled?
✅ Readability — Is the code clear without needing comments?
✅ Test coverage — Are the important paths tested?
✅ Breaking changes — Does this break existing behavior?
✅ Naming — Are functions, variables, files named clearly?
```

### Giving Good Feedback

**Be specific and actionable:**

❌ Bad: "This function is bad."

✅ Good: "This function could cause a memory leak in cases where `callback` throws an exception, because `connection` won't be closed. Consider using try-finally."

**Use prefixes to classify comments:**
```
[blocking] — Must be fixed before merge
[suggestion] — Nice to have, not required
[nit] — Minor style issues
[question] — I want to understand this better  
[praise] — This is great!
```

**Ask questions rather than demand changes:**
```
❌ "Change this to use a Map instead."
✅ "What do you think about using a Map here? It might improve lookup performance for large sets."
```

**Distinguish opinion from fact:**
```
"In my opinion, extracting this to a helper would improve readability"  
vs.
"This will cause a NullPointerException when user is null"
```

## Review Etiquette

**For reviewers:**
- Review promptly (within 24h is ideal)
- Approve when satisfied, don't nitpick indefinitely
- Mark comments as suggestions when they're not blockers
- Acknowledge improvements in follow-up reviews

**For authors:**
- Respond to all comments (even if declining)
- Don't take feedback personally
- Ask for clarification on unclear comments
- Push follow-up commits, then re-request review

## Automating Code Quality Checks

Automate what can be automated so human reviewers focus on logic:

```yaml
# .github/workflows/ci.yml
jobs:
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run lint          # ESLint
      - run: npm run type-check    # TypeScript
      - run: npm test              # Unit tests
      - run: npm run test:e2e      # End-to-end tests
      - run: npx prettier --check .  # Formatting
```

When CI is automated, reviewers can say: "CI passes, so I can focus on the design and logic, not syntax."

## CODEOWNERS

Use GitHub's CODEOWNERS to automatically assign reviewers:

```
# .github/CODEOWNERS

# Global owner for all files
* @lead-developer

# Frontend requires frontend team review
src/components/ @frontend-team

# Backend requires backend team review
src/api/ @backend-team

# Security-sensitive code requires security review
src/auth/ @security-team @lead-developer

# Database changes require DBA review
src/migrations/ @dba-team
```

## Review Metrics

Healthy code review metrics:
- Average PR size: < 400 lines
- Time to first review: < 24 hours
- PR cycle time (open to merge): < 3 days
- Review iterations: 1-2 rounds typically
