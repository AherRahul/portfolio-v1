# Git Object Storage — Inside .git/objects/

Git uses a **content-addressable filesystem** to store all data. Every piece of content has a unique identifier (SHA-1 hash) derived from its content. This design makes Git extremely reliable and efficient.

## Content-Addressable Storage

In a traditional filesystem, a file is accessed by its path (location-based). In Git, every object is accessed by its **SHA-1 hash of its content** (content-based):

```bash
# Hash any string
echo "hello world" | git hash-object --stdin
# 8c7e5a667f1b771847fe88c01c3de34413a1b220

# The hash IS the identity of "hello world" — if content is the same, hash is same
```

This means:
- Identical files are stored only **once** (deduplication)
- Any corruption changes the hash, making it **self-verifying**
- The same content has the same hash across all repositories

## The .git/objects/ Directory

```
.git/objects/
  8c/
    7e5a667f1b771847fe88c01c3de34413a1b220   ← "hello world" blob
  ab/
    c1234567890def...                         ← Another object
  pack/
    pack-….pack   ← Packed objects
    pack-….idx    ← Pack index
  info/
```

Objects are stored in subdirectories named by the first two characters of their SHA-1. This prevents any directory from having too many entries.

## The Four Object Types

### 1. Blob — File Contents

```bash
# Create a blob
echo "Hello, Git!" | git hash-object -w --stdin
# bf39de73ad7bd3d4dbb8...

# Read it back
git cat-file -p bf39de73
# Hello, Git!

git cat-file -t bf39de73
# blob
```

A blob stores **raw file contents** only — no filename, no permissions.

### 2. Tree — Directory Listing

```bash
# After staging some files:
git write-tree
# abc1234def5678...

git cat-file -p abc1234
# 100644 blob 8c7e5a... README.md
# 100644 blob bf39de... src/auth.js
# 040000 tree a1b2c3... tests/
```

A tree stores:
- File mode (100644=regular, 100755=executable, 120000=symlink, 040000=directory)
- Object type (blob or tree)
- Object SHA-1
- Filename

### 3. Commit — Snapshot Reference

```bash
git cat-file -p HEAD
# tree abc1234def5678...        ← Root tree
# parent def5678abc1234...      ← Previous commit (0 for initial)
# author Jane Doe <jane@...> 1705305600 +0000
# committer Jane Doe <jane@...> 1705305600 +0000
#
# Add authentication module
```

A commit stores:
- Root tree SHA-1 (the complete snapshot)
- Parent commit SHA-1(s)
- Author and committer info with timestamps
- Commit message

### 4. Tag — Named Commit Reference

```bash
git cat-file -p v1.0.0
# object abc1234...   ← Tagged commit
# type commit
# tag v1.0.0
# tagger Jane Doe <jane@...> 1705305600 +0000
#
# Release v1.0.0
```

## Tracing a Commit's Full Object Graph

A single commit references an entire filesystem snapshot:

```
COMMIT (abc1234)
  └── TREE (root directory)
        ├── BLOB (README.md contents)
        ├── BLOB (package.json contents)
        └── TREE (src/)
              ├── BLOB (src/auth.js contents)
              └── BLOB (src/payment.js contents)
```

```bash
# Walk the tree manually
git cat-file -p HEAD          # See commit → tree hash
git cat-file -p <tree-hash>   # See tree → children
git cat-file -p <blob-hash>   # See file contents
```

## SHA-1 and Integrity

Git verifies integrity automatically — if a file is corrupted, its SHA-1 won't match:

```bash
git fsck          # Verify all objects
git fsck --full   # More thorough check

# Output if corruption detected:
# error: sha1 mismatch for .git/objects/ab/c1234...
# error: abc1234...: object corrupt or missing
```
