# git revert Deep Dive

`git revert` is the safe way to undo changes that have been shared with others. Unlike `git reset`, which rewrites history, `git revert` creates **new commits** that undo the effects of previous commits, leaving the history intact.

## How git revert Works

`git revert` applies the **inverse** of a commit's changes:

```bash
# Revert the most recent commit
git revert HEAD

# Revert a specific commit (by hash)
git revert abc1234

# Opens editor to write/confirm the revert commit message
# Default message: "Revert 'Original commit message'"
```

What happens internally:
1. Git computes the diff of the target commit (what it added/removed)
2. Creates a new commit applying the **reverse** of that diff
3. Original commit history is preserved untouched

## Revert vs Reset vs Restore

| Command | History Preserved | Use When |
|---------|------------------|----------|
| `git revert` | ✅ Yes | Undoing pushed/shared commits |
| `git reset` | ❌ No (rewrites) | Undoing local unpushed commits |
| `git restore` | N/A (file op) | Discarding file-level changes |

**Rule of thumb:** For anything pushed to a shared branch, use `git revert`.

## Revert Options

```bash
# Revert without creating a commit (stage changes only)
git revert --no-commit abc1234
git revert -n abc1234   # Same

# This lets you bundle multiple reverts into one commit:
git revert -n abc1234
git revert -n def5678
git revert -n g9h0i1j
git commit -m "Revert three bad commits"

# Revert without opening the editor
git revert --no-edit abc1234

# Revert and sign the revert commit
git revert -S abc1234
```

## Reverting a Range of Commits

```bash
# Revert commits from def5678 to g9h0i1j (inclusive)
git revert def5678^..g9h0i1j

# The ^ includes the first commit too
# Creates a separate revert commit for each commit in range

# If you want one combined revert commit:
git revert -n def5678^..g9h0i1j
git commit -m "Revert commits for failed feature X"
```

## Reverting a Merge Commit

This is more complex. A merge commit has **two parents**, and Git needs to know which parent represents the "mainline" (the branch you merged INTO):

```bash
# Merge commit M has parent 1 (main) and parent 2 (feature branch)
git log --oneline --graph

# * merge_sha (HEAD) Merge feature into main
# |\
# | * feature_commit_2
# | * feature_commit_1
# * main_previous_commit

# Revert the merge (parent 1 = main branch = what to keep)
git revert -m 1 merge_sha
```

`-m 1` means "revert relative to parent 1" (the mainline). This effectively undoes all the feature branch changes.

> ⚠️ Warning: After reverting a merge, **re-merging the feature branch later is tricky**. Git thinks the changes have already been applied. You may need to revert the revert first.

## Handling Revert Conflicts

Conflicts can occur when the code has changed since the original commit:

```bash
git revert abc1234
# CONFLICT: auto merging src/auth.js

# 1. Look for conflict markers
git status

# 2. Edit and resolve
# The revert is trying to remove what was added in abc1234
# You need to manually apply the removal

# 3. Stage resolved files
git add src/auth.js

# 4. Continue
git revert --continue

# Or abort entirely
git revert --abort
```

## Using git revert in Practice

```bash
# Production HOTFIX: bad commit in main
git log --oneline -5 origin/main
# abc1234 Deploy: Payment system upgrade  ← This broke things!
# def5678 Add monitoring alerts

# Revert the bad deploy commit
git switch main
git pull origin main
git revert --no-edit abc1234
git push origin main   # Push the revert immediately
```

## Revert in GitHub

GitHub UI lets you revert PRs with one click:
- Go to the merged PR
- Click **"Revert"** button
- This creates a new PR that reverts all changes from the original PR

This is the cleanest approach for reverting merged PRs as it preserves the full audit trail.
