# git cherry-pick

`git cherry-pick` allows you to apply specific commits from one branch to another, without merging the entire branch. It's like saying "I want this specific commit's changes, and only this one."


## The Cherry-Pick Concept

Unlike `git merge` (which combines entire branch histories) or `git rebase` (which moves all commits to a new base), `git cherry-pick` selectively copies one or more individual commits from anywhere in the history.

```
main:       A → B → C
                     ↑ HEAD

feature:    A → B → D → E → F
                    ↑
            (we want commit D)

After cherry-pick D onto main:
main:       A → B → C → D'
```

`D'` is a new commit with the same changes as `D` but with a different parent and therefore a different SHA-1 hash.


## Basic Usage

```bash
# Cherry-pick a single commit
git cherry-pick <commit-hash>

# Cherry-pick multiple specific commits
git cherry-pick abc1234 def5678 g9h0i1j

# Cherry-pick a range of commits (exclusive of start, inclusive of end)
git cherry-pick abc1234..def5678

# Cherry-pick a range (inclusive of both)
git cherry-pick abc1234^..def5678
```


## Common Use Cases

### Applying a Hotfix to Multiple Branches

A bug was fixed in `main` and now you need that fix in `release/v1.x` too:

```bash
# Get the commit hash of the bugfix
git log --oneline main | head -5
# abc1234 Fix critical null pointer in payment processing

# Apply to release branch
git switch release/v1.x
git cherry-pick abc1234
git push origin release/v1.x
```

### Moving a Commit Made on the Wrong Branch

You accidentally committed to `main` instead of a feature branch:

```bash
# Note the commit hash
git log --oneline
# def5678 Add experimental feature (should be on feature branch)

# Create and switch to the feature branch  
git switch -c feature/experiment

# Cherry-pick the commit
git cherry-pick def5678

# Now remove it from main
git switch main
git reset --hard HEAD~1  # Remove the last commit from main
```


## Cherry-Pick Options

```bash
# Cherry-pick without creating a commit (stage changes only)
git cherry-pick --no-commit <hash>
# Allows you to review before committing

# Cherry-pick and keep the original author info
git cherry-pick --signoff <hash>

# Add a reference to the original commit in the message
git cherry-pick -x <hash>
# Adds "(cherry picked from commit abc1234)" to message
```


## Handling Cherry-Pick Conflicts

When cherry-picking causes conflicts:

```bash
git cherry-pick abc1234
# CONFLICT (content): Merge conflict in src/payment.js
```

Resolution:

```bash
# 1. Fix conflicts in the file
# 2. Stage the resolved file
git add src/payment.js

# 3. Continue the cherry-pick
git cherry-pick --continue

# Or abort if needed
git cherry-pick --abort
```


## When to Use Cherry-Pick

✅ **Good use cases:**
- Applying a hotfix to multiple release branches
- Salvaging one good commit from a branch you're about to delete
- Moving a commit made on the wrong branch
- Applying a specific feature to multiple product variations

❌ **Avoid cherry-pick when:**
- You need all changes from a branch (use merge/rebase)
- The commits have complex dependencies on each other
- You're cherry-picking large ranges of commits (use rebase)
- It would cause significant duplicate commit history


## Cherry-Pick with Multiple Commits

```bash
# Cherry-pick three specific commits (in order!)
git cherry-pick abc1234 def5678 g9h0i1j

# Cherry-pick range (does NOT include the first commit)
git cherry-pick v1.0..v2.0

# Cherry-pick range (includes BOTH boundary commits)
git cherry-pick v1.0^..v2.0
```

**Important:** Cherry-pick commits in the correct order (oldest first)! Otherwise, you may get unnecessary conflicts.
