# Git in CI/CD Pipelines

CI/CD (Continuous Integration/Continuous Deployment) is tightly coupled with your Git workflow. Every push, PR, and tag can trigger automated pipelines. Understanding this integration makes you a more effective engineer.

## The CI/CD Git Connection

```
Developer   →  git push  →  CI/CD Pipeline  →  Tests/Build  →  Deploy
             (trigger)        (automated)       (verified)      (production)
```

## Trigger Events

GitHub Actions and most CI systems support these Git-based triggers:

```yaml
on:
  push:
    branches: [main, develop]    # Push to specific branches
  pull_request:
    branches: [main]             # PR opened/updated targeting main
  push:
    tags: ['v*']                 # Tag push (releases)
  schedule:
    - cron: '0 0 * * *'         # Nightly builds
  workflow_dispatch:             # Manual trigger
```

## Branch-Based Deployment Strategy

Different branches deploy to different environments:

```
feature/*  →  (PR checks: lint + test)
develop    →  CI → Deploy to "dev" environment
main       →  CI → Deploy to "staging" → Manual approval → Deploy to "production"
```

```yaml
# .github/workflows/deploy.yml
jobs:
  deploy-dev:
    if: github.ref == 'refs/heads/develop'
    environment: development
    steps:
      - run: deploy --env=dev

  deploy-staging:
    if: github.ref == 'refs/heads/main'
    environment: staging
    steps:
      - run: deploy --env=staging

  deploy-production:
    if: github.ref == 'refs/heads/main'
    needs: [deploy-staging, integration-tests]
    environment: 
      name: production
      url: https://myapp.com
    steps:
      - run: deploy --env=prod
```

## PR-Based Workflows

Every PR should have these gates:

```yaml
on:
  pull_request:
    types: [opened, synchronize, reopened]

jobs:
  pr-checks:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0   # Full history for accurate diffs

      - name: Lint
        run: npm run lint

      - name: Unit Tests
        run: npm test

      - name: Build
        run: npm run build

      - name: Integration Tests
        run: npm run test:integration

      - name: Code Coverage
        uses: codecov/codecov-action@v4
```

## Protected Branch Rules

Configure on GitHub (Settings → Branches → Branch protection rules):

```
Branch: main
✅ Require pull request reviews (1-2 approvals)
✅ Require status checks to pass (lint, test, build)
✅ Require branches to be up to date before merging
✅ Restrict pushes to matching branches
✅ Require linear history (no merge commits)
```

## Tag-Based Releases

```yaml
on:
  push:
    tags: ['v*']

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm run build
      - run: npm test
      
      - name: Create GitHub Release
        uses: softprops/action-gh-release@v2
        with:
          generate_release_notes: true
          files: |
            dist/app-linux.tar.gz
            dist/app-macos.tar.gz
            dist/app-windows.zip
```

## GitOps — Infrastructure as Code

GitOps treats infrastructure state as code in a Git repository:

```
Git repo (desired state) → CI/CD watches for changes → Applies to infrastructure
```

```bash
# Kubernetes GitOps example with ArgoCD
# When you commit to the config repo, ArgoCD detects the change
# and synchronizes the cluster to match

# Update a deployment configuration
echo "image: myapp:v1.2.3" > k8s/deployment.yaml
git commit -am "Deploy v1.2.3 to production"
git push origin main
# ArgoCD automatically deploys the new version
```

## Rollback Strategies

```bash
# Option 1: Git revert (creates new commit, safest)
git revert abc1234   # Revert a bad commit
git push origin main
# CI/CD deploys the reverted version

# Option 2: Redeploy a previous tag
git checkout v1.1.0  # or just deploy from this tag
gh workflow run deploy.yml --ref v1.1.0

# Option 3: Cherry-pick the fix to a hotfix branch
git switch -c hotfix/critical main
git cherry-pick abc1234  # The fix
git push origin hotfix/critical
# CI/CD deploys hotfix
```

## Environment Variables and Secrets

```yaml
jobs:
  deploy:
    steps:
      - name: Deploy
        env:
          API_KEY: ${{ secrets.PROD_API_KEY }}
          DATABASE_URL: ${{ secrets.PROD_DB_URL }}
          APP_ENV: production
        run: ./deploy.sh
```

Never hardcode secrets in your code or commit history. Always use CI/CD secret management.
