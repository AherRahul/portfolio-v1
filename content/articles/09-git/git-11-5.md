# Team Git Workflows

Effective Git usage in a team context requires more than just knowing the commands — it requires agreed-upon conventions, tooling, and standards that every team member follows consistently.

## Repository Setup for Teams

### Branch Protection Rules

Configure on GitHub (Settings → Branches) for each protected branch:

```
Branch: main
✅ Require a pull request before merging
  - Required approving reviews: 1-2
  - Dismiss stale pull request approvals when new commits are pushed
  - Require review from Code Owners
✅ Require status checks to pass before merging
  - Require branches to be up to date before merging
  - Status checks: lint, test, build
✅ Require conversation resolution before merging
✅ Require signed commits
✅ Do not allow bypassing the above settings
```

### CODEOWNERS

```
# .github/CODEOWNERS

# Global owners (review all changes)

```

## Commit Message Standards

Enforce consistent commit messages with commitlint:

```bash
npm install --save-dev @commitlint/cli @commitlint/config-conventional husky
```

```js
// commitlint.config.js
module.exports = {
  extends: ['@commitlint/config-conventional'],
  rules: {
    'type-enum': [2, 'always', [
      'feat', 'fix', 'docs', 'style', 'refactor',
      'perf', 'test', 'chore', 'revert', 'ci'
    ]],
    'subject-max-length': [2, 'always', 72],
  },
};
```

```json
// package.json
{
  "husky": {
    "hooks": {
      "commit-msg": "commitlint --edit $1",
      "pre-commit": "npm run lint && npm test"
    }
  }
}
```

## Branch Naming Conventions

Agree on and document a naming convention:

```bash
# Feature work
feature/<issue-id>-<short-description>
feature/234-user-authentication
feat/456-payment-integration

# Bug fixes
fix/<issue-id>-<description>
fix/789-login-null-pointer

# Hotfixes (from main to production)
hotfix/<issue-id>-<description>
hotfix/1011-security-patch

# Documentation
docs/<description>
docs/api-reference-update

# Chores (tooling, dependencies)
chore/<description>
chore/upgrade-dependencies

# Release branches
release/v1.2.0
```

## Onboarding New Team Members

A good onboarding document covers:

```markdown
## Git Setup

1. Clone the repository
2. Configure your identity  
3. Install Husky hooks: `npm install`
4. Review our branching convention (see CONTRIBUTING.md)

## Daily Workflow

1. Start the day: `git pull origin main`
2. Create your branch: `git switch -c feature/your-task`
3. Commit regularly with conventional commits
4. Push and create PR when ready for review
5. Request review from @team-member

## Getting Help

- Ask in #git-help Slack channel
- Pair with a senior on your first PR
```

## Resolving Team Conflicts

When two developers' branches conflict:

```bash
# Sync with main first
git fetch origin
git rebase origin/main   # Rebase your branch

# Resolve any conflicts
git status   # See conflicted files
# Edit files to resolve
git add resolved-file.js
git rebase --continue

# Push (force-with-lease is safe — won't override others' pushes)
git push --force-with-lease origin feature/your-branch
```

## Regular Team Cadence

Healthy teams adopt these practices:

- **Daily:** Sync local branches with main before starting work
- **Per PR:** Code review with constructive feedback within 24h
- **Weekly:** Team retro on Git workflow pain points
- **Per Release:** Tag release, generate changelog, deploy
- **Quarterly:** Review and update team Git conventions
