# git subtree — Embedded Repositories

`git subtree` is an alternative to `git submodule` for incorporating external repositories into your project. Unlike submodules, subtrees embed the external code directly into your repository's history, making them simpler to use for most contributors.

## Subtree vs Submodule

| Aspect | git subtree | git submodule |
|--------|-------------|---------------|
| External code | Embedded in repo | Referenced separately |
| Contributor complexity | Simple (regular git usage) | Must know about submodules |
| Cloning | Normal clone works | Need `--recurse-submodules` |
| History | All in one repo | Separate repo histories |
| Updates from upstream | Manual/explicit | More explicit |
| Pushing changes upstream | Supported | Supported (in subtree) |
| Repo size | Larger (contains all code) | Smaller |

## Adding a Subtree

```bash
# Add an external repository as a subtree
git subtree add \
  --prefix=<local-directory> \
  <repository-url> \
  <branch> \
  [--squash]

# Example: add a shared components library
git subtree add \
  --prefix=src/shared-components \
  https://github.com/company/shared-components.git \
  main \
  --squash
```

The `--squash` flag merges the external repo's history into a single commit, keeping your repository history cleaner.

## Pulling Updates from Upstream

When the external repository gets updates:

```bash
# Pull latest changes from upstream
git subtree pull \
  --prefix=src/shared-components \
  https://github.com/company/shared-components.git \
  main \
  --squash
```

## Adding a Remote for Convenience

Instead of specifying the full URL every time:

```bash
# Add the remote
git remote add shared-components https://github.com/company/shared-components.git

# Now use the remote name
git subtree add --prefix=src/shared-components shared-components main --squash
git subtree pull --prefix=src/shared-components shared-components main --squash
git subtree push --prefix=src/shared-components shared-components main
```

## Pushing Changes Back to Upstream

If you make improvements to the shared code in your project and want to contribute them back:

```bash
git subtree push \
  --prefix=src/shared-components \
  shared-components \
  feature/my-improvement
```

This extracts only the commits that touched `src/shared-components/` and pushes them as a branch to the shared-components repository.

## Splitting a Directory into a Separate Repository

Convert a directory of an existing project into its own repository:

```bash
# Extract the src/utils/ directory into its own history
git subtree split --prefix=src/utils/ --branch utils-only

# Push to a new repo
git push https://github.com/company/utils.git utils-only:main
```

## Common Workflow

```bash
# Initial setup
git remote add theme https://github.com/company/theme.git
git subtree add --prefix=theme/ theme main --squash

# Daily work - theme is just a regular directory, no special commands
ls theme/   # regular files

# Periodic update from upstream
git subtree pull --prefix=theme/ theme main --squash

# Commit improvements back upstream
git subtree push --prefix=theme/ theme feature/new-colors
```

## When to Use Subtrees

✅ **Good for subtrees:**
- Teams that don't want contributors to learn submodule commands
- Embedding shared libraries where you rarely need to update
- Projects where a clean, self-contained clone is essential
- Monorepo splitting (extracting a module from a large repo)

❌ **Consider submodules instead when:**
- The external repo is very large (subtree embeds it ALL in your repo)
- You need to lock to a specific commit precisely
- The external code changes frequently and disk space matters
