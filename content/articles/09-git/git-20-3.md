# Git Performance Optimization

As repositories grow, Git operations can become slow. Understanding what causes performance issues and how to address them is essential for large projects.

## Common Performance Issues

1. **Large repository size** — Many binary files, long history
2. **Many files** — Large codebases with thousands of files
3. **Status checking slowdowns** — OS file system scanning
4. **Clone and fetch time** — Downloading large histories
5. **Many branches/tags** — Reference resolution overhead

## Shallow Clone

Download only recent history instead of the full history:

```bash
# Clone with only the last 10 commits
git clone --depth=10 https://github.com/repo.git

# Clone with only the last 1 commit (fastest)
git clone --depth=1 https://github.com/repo.git

# This is perfect for CI/CD where you only need current code
# GitHub Actions does this by default with actions/checkout

# Optionally deepen later
git fetch --deepen=50
# Or get full history
git fetch --unshallow
```

## Partial Clone (Filter)

Download only specific objects from the remote:

```bash
# Exclude blobs (file contents) until needed
git clone --filter=blob:none https://github.com/repo.git

# Exclude blobs larger than 1MB
git clone --filter=blob:limit=1m https://github.com/repo.git  

# Exclude all tree objects too (most minimal)
git clone --filter=tree:0 https://github.com/repo.git

# These are "lazy" clones — missing objects are fetched on demand
```

## Sparse Checkout

Only check out specific directories (huge projects where you need only part):

```bash
# Set up sparse checkout
git clone --no-checkout https://github.com/huge-mono-repo.git
cd huge-mono-repo
git sparse-checkout init --cone

# Only check out specific directories
git sparse-checkout set packages/my-package docs/

git checkout main  # Now only those dirs are in working tree
```

## Filesystem Monitor (fsmonitor)

On large repos, `git status` and similar commands scan all files. Enable file system monitoring for faster results:

```bash
# Enable file system monitor
git config core.fsmonitor true
git config core.untrackedCache true

# Start the background daemon
git fsmonitor--daemon start
```

## Maintenance Command

Run periodic maintenance to keep the repository fast:

```bash
# Run all maintenance tasks
git maintenance run --auto

# Run all maintenance tasks immediately
git maintenance run

# Schedule automatic maintenance
git maintenance start  # Registers a system job to run regularly
```

Maintenance includes:
- Repacking objects (pack-refs, loose-objects, incremental-repack)
- Committing graph updates (commit-graph)
- Prefetching remotes

## Commit Graph

The commit graph file speeds up operations that traverse commit history:

```bash
# Build the commit graph
git commit-graph write --reachable

# Build incrementally (faster for large repos)
git commit-graph write --reachable --changed-paths --split

# Enable commit graph
git config fetch.writeCommitGraph true
```

## Repository Cleanup

```bash
# Run garbage collection (pack loose objects, prune old refs)
git gc

# Aggressive repack (slower, but better compression)
git gc --aggressive

# Prune unreachable objects (older than 2 weeks)
git prune

# Prune immediately
git gc --prune=now

# Verify repository integrity
git fsck
```

## Large File Handling

```bash
# Find the largest objects in history
git rev-list --objects --all | \
  git cat-file --batch-check='%(objecttype) %(objectname) %(objectsize) %(rest)' | \
  awk '/^blob/ {print substr($0,6)}' | \
  sort --numeric-sort --key=2 | \
  tail -10

# Move them to Git LFS
git lfs migrate import --include="*.mp4,*.psd,*.zip"
```

## CI/CD Performance Tips

```yaml
# .github/workflows/ci.yml
steps:
  - uses: actions/checkout@v4
    with:
      fetch-depth: 1    # Shallow clone for CI
      
  - uses: actions/cache@v4
    with:
      path: ~/.npm
      key: ${{ runner.os }}-npm-${{ hashFiles('**/package-lock.json') }}
```

Always use shallow clones and caching in CI environments to reduce build times dramatically.
