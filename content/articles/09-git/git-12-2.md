# Annotated vs Lightweight Tags

Git has two types of tags: **lightweight** and **annotated**. Understanding the difference helps you choose the right type for your use case.

## Lightweight Tags

A lightweight tag is simply a named pointer to a specific commit — essentially an alias for a commit hash. It stores no additional information.

```bash
# Create a lightweight tag
git tag v1.0.0

# View it
git show v1.0.0
# Output: Shows the commit directly (not a tag object)
```

**Characteristics:**
- Just a name pointing to a commit
- No tagger info, no date, no message
- Not a "real" Git object (no SHA-1 of its own)
- Appears in `git log` decorations

**When to use lightweight tags:**
- Personal bookmarks while working
- Temporary "save point" tags
- Scripts that only need a named reference

## Annotated Tags

An annotated tag is a full Git object with its own SHA-1. It stores:
- The tag name
- Tagger name and email
- Tagging date
- Tag message
- Optional GPG signature

```bash
# Create an annotated tag
git tag -a v1.0.0 -m "Release version 1.0.0"

# View the tag object
git show v1.0.0
# Output:
# tag v1.0.0
# Tagger: Jane Doe <jane@example.com>
# Date:   Mon Jan 15 2024 10:00:00
#
# Release version 1.0.0
#
# commit abc1234...
# Author: Jane Doe...
# ...
```

**Characteristics:**
- Full Git object with its own SHA-1
- Contains tagger metadata
- Can be GPG-signed
- Can be transferred with `git fetch`/`git push` separately from commits

## Key Differences

| Feature | Lightweight | Annotated |
|---------|------------|-----------|
| Git object | No (alias) | Yes (own SHA-1) |
| Tagger info | No | Yes |
| Tagging date | No | Yes |
| Message | No | Yes |
| GPG signature | No | Yes |
| Use for releases | ❌ | ✅ |
| `git describe` | Only if annotated nearby | Yes |

## git describe Behavior

```bash
# git describe only considers annotated tags by default
git tag lightweight-tag     # lightweight
git describe HEAD
# fatal: No names found, cannot describe anything

git tag -a v1.0.0 -m "msg"  # annotated
git describe HEAD
# v1.0.0                    # Works!

# For lightweight:
git describe --tags HEAD
# v1.0.0 or lightweight-tag
```

## Recommendation

**Always use annotated tags for releases.** Use the `-a` flag with a meaningful message:

```bash
# Good release tag
git tag -a v2.1.0 -m "Release v2.1.0: Add payment integration

Changelog:
- feat: Stripe payment gateway
- fix: Handle declined cards
- docs: Update API documentation"

git push origin v2.1.0
```

Annotated tags provide the metadata future developers and tools need to understand when and why a release was made.

## Listing Tag Types

```bash
# See all tags
git tag

# See if tag is annotated (look for "tag" in type)
git cat-file -t v1.0.0
# tag       ← annotated (its own object)
# commit    ← lightweight (points directly to commit)
```
