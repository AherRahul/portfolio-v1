# GitHub Releases — Publishing Versions

GitHub Releases are a way to package and publish software to users. They build on Git tags and add a user-friendly interface with release notes, downloadable assets, and version history.

## What is a Release?

A GitHub Release:
- Is **linked to a Git tag** (e.g., `v1.2.0`)
- Has a **title** and **release notes** (changelog)
- Can contain **binary assets** (compiled binaries, archives)
- Can be marked as **latest** or **pre-release**
- Has a **permalink** (`github.com/owner/repo/releases/tag/v1.2.0`)

## Creating a Release on GitHub

1. Go to your repository
2. Click **"Releases"** in the right sidebar
3. Click **"Draft a new release"** or **"Create a new release"**
4. Choose or create a **tag** (e.g., `v1.2.0`)
5. Fill in:
   - **Release title** (e.g., `v1.2.0 — Payment Integration`)
   - **Release notes** (describe what changed)
   - **Attach binaries** if applicable
   - Check **Pre-release** if it's alpha/beta
6. Click **"Publish release"**

## Writing Release Notes

Good release notes include:

```markdown
## What's New in v2.0.0

### ✨ New Features
- Added OAuth2 Google and GitHub login (#123)
- New dashboard with real-time analytics (#145)
- Dark mode support (#178)

### 🐛 Bug Fixes
- Fix null pointer exception in payment processing (#156)
- Fix session expiry not working on mobile (#162)

### 💥 Breaking Changes
- Renamed `getUserData()` to `fetchUser()` in the API
- The `/api/v1/users` endpoint now returns paginated results

### 📦 Dependencies Updated
- Upgraded to Node.js 20 LTS
- Stripe SDK updated to v14

**Full Changelog:** https://github.com/user/project/compare/v1.5.0...v2.0.0
```

## Release from the Command Line

First create a tag:

```bash
git tag -a v1.2.0 -m "Release v1.2.0"
git push origin v1.2.0
```

Then create the release on GitHub.com, or use the **GitHub CLI**:

```bash
# Install GitHub CLI (gh)
gh release create v1.2.0 --title "v1.2.0 — Bug Fixes" --notes "Fixed important bugs"

# Create a pre-release
gh release create v2.0.0-beta.1 --prerelease --notes "Beta release"

# Attach a file to the release
gh release create v1.2.0 --notes "..." dist/app-linux-amd64.tar.gz
```

## Automating Releases with GitHub Actions

```yaml
# .github/workflows/release.yml
name: Create Release

on:
  push:
    tags:
      - 'v*'

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build
        run: npm run build

      - name: Create Release
        uses: softprops/action-gh-release@v2
        with:
          generate_release_notes: true  # Auto-generate from PRs
          files: dist/**
```

## Pre-releases vs Stable Releases

| Type | Use | Example |
|------|-----|---------|
| **Stable** | Production-ready | `v1.2.0` |
| **Pre-release (alpha)** | Early feature testing | `v2.0.0-alpha.1` |
| **Pre-release (beta)** | More complete testing | `v2.0.0-beta.3` |
| **Release candidate** | Final testing | `v2.0.0-rc.1` |

## Best Practices

1. **Every release should have a tag** — meaningful version number (SemVer).
2. **Write comprehensive release notes** — users need to know what changed.
3. **Include migration instructions** for breaking changes.
4. **Link to the full changelog** for detailed diffs.
5. **Automate releases** with GitHub Actions triggered on tag push.
6. **Test before releasing** — never publish broken code.
