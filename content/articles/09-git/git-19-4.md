# Changelog Generation

A good changelog communicates what changed between releases to users and developers. With conventional commits and the right tools, changelog generation can be fully automated from your Git history.

## Why Automated Changelogs?

Manual changelogs are:
- Tedious to write
- Often forgotten or delayed
- Inconsistent in format

Automated changelogs generated from git commit messages are:
- Always up-to-date
- Consistent format
- Linked to the actual code changes (commit hashes)

## Prerequisite: Conventional Commits

Automated tools parse commit messages. They work best with **Conventional Commits** format:

```
type(scope): subject

feat(auth): add Google OAuth login
fix(payment): handle declined card error  
docs(api): add endpoint documentation
refactor(db): simplify connection pooling
chore(deps): upgrade stripe to v14
BREAKING CHANGE: rename getUserData to fetchUser
```

## Tool 1: git-cliff (Fast, Config-Friendly)

```bash
# Install
brew install git-cliff   # macOS
# or
cargo install git-cliff  # Rust

# Generate changelog for all history
git cliff

# Generate since a tag
git cliff v1.0.0..HEAD

# Generate and write to CHANGELOG.md
git cliff -o CHANGELOG.md
```

`cliff.toml` (configuration):

```toml
[changelog]
header = """
# Changelog\n
"""
body = """
{% for group, commits in commits | group_by(attribute="group") %}
### {{ group | upper_first }}\n
{% for commit in commits %}
- {{ commit.message | upper_first }} ([{{ commit.id | truncate(length=7, end="") }}](https://github.com/user/repo/commit/{{ commit.id }}))\
{% endfor %}
{% endfor %}\n
"""
```

## Tool 2: Release Please (GitHub Actions)

Google's `release-please` automates the entire release cycle:

1. Analyzes commits since last release
2. Creates/updates a "Release PR" with version bump + changelog
3. When the Release PR is merged, creates a GitHub Release

```yaml
# .github/workflows/release-please.yml
on:
  push:
    branches: [main]

permissions:
  contents: write
  pull-requests: write

jobs:
  release-please:
    runs-on: ubuntu-latest
    steps:
      - uses: googleapis/release-please-action@v4
        with:
          token: ${{ secrets.GITHUB_TOKEN }}
          release-type: node
```

## Tool 3: standard-version

```bash
npm install --save-dev standard-version

# Analyze commits, bump version, update CHANGELOG.md, create git tag
npx standard-version

# Just dry run (preview)
npx standard-version --dry-run

# Force minor bump
npx standard-version --release-as minor

# Pre-release
npx standard-version --prerelease alpha
```

## Tool 4: semantic-release (Full Automation)

semantic-release automates everything: version bumping, changelog, GitHub releases, and npm publishing.

```bash
npm install --save-dev semantic-release \
  @semantic-release/git \
  @semantic-release/github \
  @semantic-release/changelog
```

`.releaserc.json`:
```json
{
  "branches": ["main"],
  "plugins": [
    "@semantic-release/commit-analyzer",
    "@semantic-release/release-notes-generator",
    "@semantic-release/changelog",
    "@semantic-release/npm",
    "@semantic-release/github",
    ["@semantic-release/git", {
      "assets": ["CHANGELOG.md", "package.json"],
      "message": "chore(release): ${nextRelease.version}"
    }]
  ]
}
```

```yaml
# .github/workflows/release.yml
on:
  push:
    branches: [main]

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Full history for semantic analysis
          persist-credentials: false
      - uses: actions/setup-node@v4
      - run: npm ci
      - run: npx semantic-release
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
```

## Manual Changelog from git log

If you prefer manual:

```bash
# Generate CHANGELOG for v1.0.0 → v2.0.0
git log v1.0.0..v2.0.0 --format="- %s (%h)" --no-merges | \
  grep -E "^- (feat|fix|docs|refactor|perf)" | \
  sort

# Group by type manually
echo "### Features"
git log v1.0.0..v2.0.0 --format="- %s (%h)" | grep "^- feat"

echo "### Bug Fixes"
git log v1.0.0..v2.0.0 --format="- %s (%h)" | grep "^- fix"
```

## CHANGELOG.md Format

Standard `CHANGELOG.md` format (Keep a Changelog convention):

```markdown
# Changelog

All notable changes to this project will be documented here.

## [Unreleased]

## [2.0.0] - 2024-01-15

### Breaking Changes
- Renamed `getUserData` API to `fetchUser`

### Added
- Google OAuth login support
- Dark mode across all pages

### Fixed
- Login timeout on mobile devices
- CSV export truncating large datasets

## [1.5.0] - 2023-11-20
...

[Unreleased]: https://github.com/user/project/compare/v2.0.0...HEAD
[2.0.0]: https://github.com/user/project/compare/v1.5.0...v2.0.0
[1.5.0]: https://github.com/user/project/compare/v1.0.0...v1.5.0
```
