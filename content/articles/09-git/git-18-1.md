# Semantic Commit Messages

A good commit message tells the story of your change. **Semantic commit messages** (also called Conventional Commits) provide a structured format that makes commit messages machine-readable and human-friendly, enabling automatic changelog generation, semantic versioning, and better project communication.


## The Conventional Commits Specification

The format is:

```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```


## Types

| Type | Description |
|------|-------------|
| `feat` | A new feature |
| `fix` | A bug fix |
| `docs` | Documentation changes only |
| `style` | Formatting, missing semicolons, etc. (no code change) |
| `refactor` | Code change that neither fixes a bug nor adds a feature |
| `test` | Adding or modifying tests |
| `chore` | Build process, tooling changes, dependencies |
| `perf` | Performance improvements |
| `ci` | CI/CD configuration changes |
| `build` | Build system changes |
| `revert` | Reverts a previous commit |


## Scope

The scope specifies part of the codebase affected (optional but helpful):

```
feat(auth): add Google OAuth login
fix(payment): handle NULL card number
docs(api): update REST endpoint documentation
refactor(database): extract query builder class
```


## Breaking Changes

Indicate breaking changes with:

1. A `!` after the type/scope: `feat!: changed API response format`
2. Or a `BREAKING CHANGE:` footer:

```
feat(api)!: change user endpoint response structure

BREAKING CHANGE: The /api/users endpoint now returns paginated results.
Old format: { users: [...] }
New format: { data: [...], total: N, page: N, limit: N }
```


## Complete Examples

### Simple commit

```
fix(auth): resolve token expiration bug
```

### With body

```
feat(payment): add Stripe payment integration

Integrate Stripe as the payment processor. Supports:
- Credit/debit card payments
- Recurring subscriptions
- Webhook handling for payment events

Closes #234
```

### With breaking change

```
feat(api)!: update user endpoint to return paginated data

BREAKING CHANGE: The /api/users endpoint now returns paginated results.

Before: GET /api/users → { users: [...] }
After: GET /api/users → { data: [...], total: N, page: N }

Update all API consumers to use the new paginated format.
```


## Why Use Conventional Commits?

1. **Automatic changelog generation:** Tools like `conventional-changelog` can generate CHANGELOG.md automatically.

2. **Semantic versioning automation:** Tools can determine whether a release should be MAJOR, MINOR, or PATCH based on commit types.
   - `feat` → MINOR version bump
   - `fix` → PATCH version bump
   - `BREAKING CHANGE` → MAJOR version bump

3. **Better search and filtering:**
   ```bash
   git log --grep="^feat" --oneline   # Find all feature commits
   git log --grep="^fix" --oneline    # Find all bug fixes
   ```

4. **Code review context:** Reviewers immediately understand what a PR is doing.

5. **Team communication:** Developers can find relevant changes without reading every commit.


## Enforcing Conventional Commits

Use a `commit-msg` Git hook (or Husky for Node.js projects):

```bash
# Install commitlint
npm install --save-dev @commitlint/cli @commitlint/config-conventional

# commitlint.config.js
module.exports = { extends: ['@commitlint/config-conventional'] };

# .husky/commit-msg
#!/usr/bin/env sh
. "$(dirname -- "$0")/_/husky.sh"
npx commitlint --edit "$1"
```

This will reject commit messages that don't follow the conventional format.
