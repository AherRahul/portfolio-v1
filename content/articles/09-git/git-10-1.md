# GitHub Pull Requests

A **Pull Request (PR)** is a GitHub feature that lets developers notify team members about changes they've pushed to a branch in a repository. It's the central mechanism for code review and collaboration on GitHub.


## What is a Pull Request?

A pull request:
1. **Notifies** your team that you've made changes and want them reviewed.
2. **Shows** a diff of the changes — exactly what was added, modified, or removed.
3. **Enables discussion** — team members can comment on specific lines of code.
4. **Tracks review status** — approvals, change requests, and conversations.
5. **Integrates CI/CD** — automatic tests and checks run against the PR.
6. **Merges changes** into the target branch after approval.


## Creating a Pull Request

### Steps in GitHub

1. Push your feature branch to GitHub:
   ```bash
   git push -u origin feature/login
   ```

2. Go to the repository on GitHub.

3. GitHub shows a banner: **"feature/login had recent pushes... Compare & pull request"** — click it.

4. Fill in the PR form:
   - **Title**: Concise description of the change
   - **Description**: Detailed explanation, screenshots if UI changes, issue references
   - **Reviewers**: Team members to review the PR
   - **Assignees**: Usually the author
   - **Labels**: (feature, bugfix, documentation, etc.)
   - **Milestone/Project**: For project tracking

5. Click **"Create pull request"**.


## Writing a Good PR Description

A good PR description should include:

```markdown
## What does this PR do?
Adds user authentication with OAuth2 via Google sign-in.

## Why is this change needed?
Users requested social login to reduce friction in the signup process.

## How was this tested?
- Unit tests added in tests/auth.test.js
- Manual testing with test Google accounts
- CI pipeline passes all checks

## Screenshots
[Before/After screenshots if UI change]

## Related Issues
Closes #234
Fixes #235
```


## Reviewing a Pull Request

As a reviewer:

1. **Read the PR description** to understand the context.
2. Click **"Files changed"** to see the diff.
3. **Comment on specific lines** by clicking the `+` button next to a line.
4. **Suggest changes** directly in the diff using GitHub's suggestion feature.
5. When done, submit your review as:
   - **Comment**: General feedback, no approval
   - **Approve**: Ready to merge
   - **Request changes**: Changes needed before merging


## Merging a Pull Request

GitHub offers three merge options:

| Option | Description | When to use |
|--------|-------------|-------------|
| **Create a merge commit** | Keeps all commits + adds a merge commit | Feature branches with meaningful commit history |
| **Squash and merge** | Combines all PR commits into one | WIP commits that should be one logical change |
| **Rebase and merge** | Replays commits linearly, no merge commit | For linear history purity |


## Best Practices for PRs

1. **Keep PRs small:** Large PRs are harder to review. Aim for < 400 lines changed.

2. **One PR per feature/fix:** Don't combine unrelated changes.

3. **Respond to reviews promptly:** Keep the conversation moving.

4. **Don't push to another person's PR without asking.**

5. **Resolve all conversations** before merging.

6. **Link to related issues:** `Closes #123` automatically closes the issue on merge.

7. **Test your PR locally** before requesting review.


## PR Workflow Summary

```bash
# 1. Create feature branch
git switch -c feature/oauth-login

# 2. Work and commit
git add .
git commit -m "Add Google OAuth integration"

# 3. Push to GitHub
git push -u origin feature/oauth-login

# 4. Create PR on GitHub (with good description)

# 5. Address review feedback
git add .
git commit -m "Fix Review: Update error handling"
git push

# 6. After PR is merged, clean up
git switch main
git pull  # Update local main
git branch -d feature/oauth-login  # Delete local branch
```
