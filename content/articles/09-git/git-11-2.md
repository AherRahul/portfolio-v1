# GitHub Flow — Simple, Powerful Branching

GitHub Flow is a lightweight branching strategy popularized by GitHub. It's simpler than Gitflow and works exceptionally well for continuous deployment environments where you deploy frequently.

## The Six Steps of GitHub Flow

### 1. Create a Branch

Always branch from `main`. Give it a descriptive name:

```bash
git switch main
git pull origin main
git switch -c feature/user-authentication
# Prefixes: feature/, fix/, docs/, chore/
```

### 2. Add Commits

Make small, focused commits as you work:

```bash
git add src/auth.js
git commit -m "feat: add JWT token generation"

git add tests/auth.test.js
git commit -m "test: add JWT token tests"
```

### 3. Open a Pull Request

Push your branch and open a PR **early** — even before it's done:

```bash
git push -u origin feature/user-authentication
```

On GitHub: Open a PR → Add `[WIP]` or **Draft PR** if work in progress.

PRs serve as:
- Code review tool
- Discussion thread
- Integration check point
- Documentation of what changed

### 4. Discuss and Review

Team members review the PR:
- Comment on specific lines
- Approve or request changes
- GitHub Actions CI runs automatically

### 5. Deploy from the Branch (Optional)

Deploy and test the feature branch in a staging environment before merging:

```bash
# GitHub Actions can deploy branches to staging on PR events
# After testing on staging, confirm the PR is ready to merge
```

### 6. Merge into Main

Once approved and tests pass:
- **Squash and merge** for clean history (recommended)
- **Create a merge commit** to preserve all commits
- Main is now deployable — deploy immediately

```bash
# After merge, clean up
git switch main
git pull origin main
git branch -d feature/user-authentication
```

## Visual GitHub Flow

```
main:    ───────────────────────────────●──── (always deployable)
                                       /
feature: ───●──●──●──●── (PR → review → merge)
```

## What Makes GitHub Flow Work

1. **main is always deployable** — The most important rule. main is always production-ready.

2. **Short-lived branches** — Features should be developed quickly (days, not weeks) to minimize merge conflicts.

3. **Frequent deployment** — Deploy to production often (even multiple times per day).

4. **Mandatory code review** — Every change goes through PR review before merging.

5. **CI/CD gates** — Automated tests must pass before any merge.

## GitHub Flow vs Gitflow

| Aspect | GitHub Flow | Gitflow |
|--------|------------|---------|
| Branches | `main` + feature branches | `main`, `develop`, feature, release, hotfix |
| Complexity | Simple | Complex |
| Deployment | Continuous (every merge) | Scheduled releases |
| Best for | Web apps, SaaS, continuous delivery | Versioned products, apps |
| Team size | Any | Medium to large |

## Combining with Feature Flags

For large features that take time, use feature flags to merge to main incrementally:

```javascript
if (featureFlags.newAuthSystem) {
  return newAuthHandler(req, res);
}
return legacyAuthHandler(req, res);
```

This allows merging incomplete features to `main` safely while they're disabled for users.
