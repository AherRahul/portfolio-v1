# git gc and Maintenance

Git repositories can accumulate "garbage" — loose objects, expired reflog entries, and unreachable commits — over time. `git gc` (garbage collection) and the newer `git maintenance` command clean up and optimize your repository.

## What is Garbage Collection?

Git's garbage collection:
1. **Packs loose objects** into packfiles (more efficient storage)
2. **Removes orphaned/unreachable objects** that have expired
3. **Compresses packfiles** using delta compression
4. **Cleans up stale remote tracking refs** and other references

## git gc

```bash
# Standard garbage collection
git gc

# More aggressive repack (slower, better compression)
git gc --aggressive

# Auto mode — only runs if enough loose objects
git gc --auto

# Run immediately, pruning all unreachable objects now instead of after 2 weeks
git gc --prune=now

# Quiet (silent) mode
git gc --quiet
```

### What git gc Outputs

```
Enumerating objects: 1847, done.
Counting objects: 100% (1847/1847), done.
Delta compression using up to 8 threads
Compressing objects: 100% (823/823), done.
Writing objects: 100% (1847/1847), done.
Total 1847 (delta 512), reused 0 (delta 0), pack-reused 0
```

## The Object Expiry Window

Loose unreachable objects are not immediately deleted. Git waits:
- **14 days** (default): For objects in the reflog
- **30 days** (unreachable objects): Objects not in reflog

This safety window allows recovery via `git reflog`.

```bash
# Change the safety window
git config gc.reflogExpire "30 days"
git config gc.reflogExpireUnreachable "7 days"

# Never expire reflog (for important repos)
git config gc.reflogExpire "never"
```

## git maintenance (Modern Approach)

`git maintenance` is a newer, more granular approach that separates different maintenance tasks:

```bash
# Run all maintenance tasks
git maintenance run

# Schedule maintenance to run automatically in background
git maintenance start    # Registers with OS task scheduler
git maintenance stop     # Unregister

# Run specific tasks only
git maintenance run --task=gc
git maintenance run --task=commit-graph
git maintenance run --task=fetch
git maintenance run --task=loose-objects
git maintenance run --task=incremental-repack
git maintenance run --task=pack-refs
```

### Available Tasks

| Task | Description |
|------|-------------|
| `gc` | Full garbage collection |
| `commit-graph` | Write commit graph for faster history traversal |
| `fetch` | Prefetch from remotes |
| `loose-objects` | Pack loose objects into packfiles |
| `incremental-repack` | Incrementally repack loose objects |
| `pack-refs` | Pack loose refs into a single refs file |

## Diagnosing Repository Health

```bash
# Count loose objects vs packed objects
git count-objects -v

# Output:
# count: 20          ← Loose objects (normal)
# size: 80           ← Loose objects size (KB)
# in-pack: 2341      ← Objects in packfiles
# packs: 1           ← Number of packfiles
# size-pack: 1247    ← Packfile size (KB)
# prune-packable: 0  ← Duplicate loose objects that can be pruned
# garbage: 0         ← Files that aren't .pack or .idx

# Verify repository integrity
git fsck
git fsck --full    # More thorough, checks all objects

# See what's taking space
git count-objects -vH   # Human readable sizes
```

## When to Run git gc

Git runs `git gc --auto` automatically in the background during:
- `git commit`
- `git merge`
- `git rebase`
- `git push` (on server-side)

You might manually run it if:
- Repository feels slower than usual
- You deleted many branches (their commits become unreachable)
- You ran `git filter-repo` (leaves many loose objects)
- `git count-objects` shows many thousands of loose objects

## Automatic gc Configuration

```bash
# Threshold at which git gc --auto runs (0 = disabled)
git config gc.auto 6700          # Default: 6700 loose objects

# Threshold for pack files before repacking
git config gc.autoPackLimit 50   # Default: 50 packs

# Run gc in background (don't block user)
git config gc.autoDetach true    # Default: true on Linux/Mac
```
