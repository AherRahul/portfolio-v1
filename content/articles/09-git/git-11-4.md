# Open Source Contribution Workflow

Contributing to open source projects requires a specific Git workflow that differs from internal team development. This guide covers the complete process from finding a project to getting your PR merged.

## Phase 1: Getting Started

### Fork the Repository

1. Go to the project on GitHub
2. Click **"Fork"** → This creates a copy of the repo under your account

```bash
# Clone YOUR fork (not the original)
git clone https://github.com/YOUR-USERNAME/project.git
cd project

# Add the original as "upstream"
git remote add upstream https://github.com/ORIGINAL-OWNER/project.git

# Verify your remotes
git remote -v
# origin    https://github.com/YOUR-USERNAME/project.git (your fork)
# upstream  https://github.com/ORIGINAL-OWNER/project.git (original)
```

### Explore the Repository

Before contributing:
- Read `CONTRIBUTING.md` — project-specific contribution guidelines
- Read `CODE_OF_CONDUCT.md` — community behavior expectations
- Look at open issues (find `good first issue` labels)
- Look at existing PRs to understand the style

## Phase 2: Making Your Contribution

### Keep Your Fork in Sync

Before starting new work, always sync with upstream:

```bash
# Fetch latest from upstream
git fetch upstream

# Update your local main
git switch main
git merge upstream/main   # or rebase
git push origin main      # Keep your fork's main updated too
```

### Create a Feature Branch

**Never commit directly to main in your fork:**

```bash
git switch main
git switch -c fix/typo-in-readme
# or
git switch -c feat/add-dark-mode
```

### Make Small, Focused Changes

```bash
# Make changes
# Write tests
# Update documentation if needed

git add -p   # Stage carefully, review each hunk
git commit -m "fix: correct typo in installation docs"
```

## Phase 3: Submitting Your Contribution

### Push to Your Fork

```bash
git push origin fix/typo-in-readme
```

### Create a Pull Request

On GitHub, you'll see a prompt to create a PR. Fill in:

```markdown
## What does this PR do?
Fixes a typo in the installation documentation (README.md, line 45).

"installtion" → "installation"

## Why?
Noticed this while following the setup guide.

## Type of change
- [x] Bug fix / Documentation fix
- [ ] New feature
- [ ] Breaking change
```

**PR Best Practices:**
- Reference the related issue: `Closes #123` or `Related to #123`
- Keep it small and focused
- Add screenshots for UI changes
- Ensure CI passes

## Phase 4: Handling Review Feedback

```bash
# Make requested changes
git add src/auth.js
git commit -m "Address review: add null check for user parameter"

# Push updates (same branch, same PR)
git push origin fix/typo-in-readme
```

**Communication Tips:**
- Thank reviewers for their time
- Ask for clarification if feedback is unclear
- If you disagree, explain your reasoning professionally
- Mark conversations as resolved after addressing them

## Phase 5: Staying Updated During Long PRs

If your PR takes time to get reviewed, the upstream may diverge:

```bash
# Sync with upstream
git fetch upstream
git rebase upstream/main  # Rebase your branch onto latest main

# Resolve any conflicts
git push --force-with-lease origin fix/my-feature
```

## Community Etiquette

✅ **Do:**
- Be patient — maintainers are often volunteers
- Follow the existing code style
- Write clear commit messages and PR descriptions
- Respond to feedback promptly
- Break large changes into smaller PRs

❌ **Don't:**
- Submit massive PRs without discussing first
- Ignore the contributing guidelines
- Get defensive about feedback
- Spam maintainers if your PR isn't reviewed immediately
- Make changes unrelated to your PR's scope

## After Your PR is Merged

```bash
# Sync your fork's main with the newly merged changes
git fetch upstream
git switch main
git merge upstream/main
git push origin main

# Delete your feature branch
git branch -d fix/typo-in-readme
git push origin --delete fix/typo-in-readme

# You're now a contributor! 🎉
```

## Finding First Contributions

Good starting points:
- **GitHub:** Search `label:good-first-issue label:help-wanted`
- **goodfirstissue.dev** — Curated list of beginner-friendly issues
- **up-for-grabs.net** — Issues flagged for first-time contributors
- **firsttimersonly.com** — Issues specifically for first-time contributors
