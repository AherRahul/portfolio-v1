# Debugging with Git

Git provides powerful tools for debugging — not just for version control, but for finding the source of bugs, understanding code evolution, and pinpointing exactly when something broke.

## The Debugging Mindset with Git

Every bug has a point in history when it was introduced. Git's complete history makes it possible to find that exact moment, understand the context, and fix it correctly.

## Tool 1: git bisect (Binary Search for Bugs)

The most powerful debugging tool in Git — automatically finds the commit that introduced a bug:

```bash
# 1. Start bisect
git bisect start

# 2. Mark current state as broken
git bisect bad

# 3. Mark a known good point
git bisect good v1.0.0  # Or any known-good commit hash

# 4. Git checks out the middle commit
# → Test the application
# → Run your specific reproduction case

# 5. Tell Git the result
git bisect good  # If bug is NOT present here
git bisect bad   # If bug IS present here

# 6. Repeat until Git identifies the culprit
# "abc1234 is the first bad commit"

# 7. Done — investigate that commit
git show abc1234

# 8. Clean up
git bisect reset
```

## Automated bisect

```bash
git bisect start
git bisect bad HEAD
git bisect good v2.0.0

# Automated with a test script (exits 0 = good, non-zero = bad)
git bisect run npm test
git bisect run python tests/regression_test.py
```

## Tool 2: git log for Change Investigation

```bash
# What changed in this file recently?
git log --oneline -- src/payment.js

# Show changes to a function over time
git log -L :processPayment:src/payment.js

# Find the commit that introduced a string
git log -S "calculateTax" --patches

# What was the last change that touched line 45?
git blame src/payment.js -L 45,45

# Changes in last week that touched auth files
git log --since="1 week ago" -- src/auth/
```

## Tool 3: git blame for Line-Level Tracking

```bash
# Who last touched each line and when
git blame -L 100,120 src/payment.js

# Blame at a specific commit
git blame abc1234 -- src/payment.js

# Ignore whitespace-only changes  
git blame -w src/payment.js

# Show the commit that moved this code (detect copy-paste)
git blame -CCC src/payment.js
```

## Tool 4: git stash for Isolating Bugs

```bash
# Bug present in your working changes? Stash and test
git stash  # Stash your changes
# Test the app — bug still present? It's in committed code
git stash pop  # Restore your changes
```

## Tool 5: git diff for Change Analysis

```bash
# What changed between working version and broken version?
git diff v1.0.0 v1.1.0 -- src/

# What changed in a specific file between two commits?
git diff abc1234 def5678 -- src/payment.js

# What's different between my branch and main?
git diff main..feature/broken-thing

# Summary of changes
git diff --stat v1.0.0 HEAD
```

## Debugging Workflow

```bash
# Step 1: Identify the scope
git log --oneline --since="1 week ago" -- src/auth/

# Step 2: Find the breaking commit with bisect
git bisect start
git bisect bad HEAD
git bisect good main@{1.week.ago}
# run tests after each checkout

# Step 3: Investigate the culprit
git show abc1234              # See the full diff
git log abc1234 -1            # See the commit message
git diff abc1234^ abc1234     # See exact changes

# Step 4: Understand context
git blame src/auth.js -L 45,55  # See who changed those lines
git log --all --follow -- src/auth.js  # Full history

# Step 5: Fix and verify
git revert abc1234  # Or create a fix commit
git test  # Verify the fix
```

## Finding Regressions in Production

```bash
# 1. Identify when it was last working (use deploy tags or SHAs)
# 2. Identify broken state (latest commit, or tag)
git bisect start
git bisect good v1.5.2  # Last release that worked
git bisect bad v1.6.0   # First release that's broken

# 3. Git narrows it down in ~log2(n) steps
# 4. Deploy the culprit commit to staging for verification
git checkout abc1234
# Test... confirmed broken!

# 5. Fix
git switch main
git revert abc1234
```
