# Branching Strategies

A branching strategy is a convention for how and when branches are created and merged in a project. Choosing the right strategy for your team significantly impacts your workflow efficiency, code quality, and release stability.


## 1. Trunk-Based Development

The simplest strategy: everyone commits to a single branch (often called `main` or `trunk`). Feature flags control which features are released.

```
main: ─────●─────●─────●─────●───── (everyone commits here)
```

**Best for:**
- Small teams with high trust
- Continuous deployment environments
- Projects with excellent automated testing

**Key practices:**
- Very short-lived feature branches (< 1 day)
- Feature flags for incomplete features
- Comprehensive CI/CD pipeline

**Pros:** Simple, enables continuous integration
**Cons:** Requires discipline and excellent test coverage


## 2. GitHub Flow

A lightweight, single-branch model where all feature work happens on branches that are merged via pull requests into `main`.

```
main:    ─────●─────────────────●───── (stable, deployable)
               \               /
feature:        └─ feature ───┘
```

**Best for:**
- Web applications with continuous deployment
- Small to medium teams
- Projects that deploy frequently

**Workflow:**
1. Branch from `main`
2. Add commits
3. Open a pull request
4. Discuss and review
5. Deploy for testing
6. Merge to `main`

**Pros:** Simple, great for web apps, easy to understand
**Cons:** Not ideal for versioned releases or supporting multiple versions


## 3. Gitflow

A structured model with separate branches for features, releases, and hotfixes (covered in the previous chapter).

```
main:    ───●───────────●──── (tagged releases)
             \         /
develop: ─────●─●───●─● ─── (integration)
                    │
feature:        ────┘ 
```

**Best for:**
- Projects with scheduled releases
- Multiple production versions to support
- Need for formal release management

**Pros:** Great for versioned releases, clear branch structure
**Cons:** Complex, overhead for small teams or continuous deployment


## 4. Release Flow (Microsoft)

Used by large companies like Microsoft. Features branch from `main`, and releases are cut as branches from a stable state.

```
main:    ─────●─────●─────●─────
release:           └── release/1.0
                        └── hotfixes merged in
```


## 5. Feature Branch Workflow

Features are always developed in dedicated branches, but there's no `develop` branch — features merge directly to `main`.

**Best for:** Most medium-sized teams


## Choosing the Right Strategy

| Strategy | Team Size | Deployment | Complexity | Best For |
|----------|-----------|------------|------------|---------|
| Trunk-based | Small | Continuous | Low | Startups, lean teams |
| GitHub Flow | Small-Medium | Continuous | Low | Web apps, SaaS |
| Gitflow | Medium-Large | Scheduled | High | Enterprise, versioned products |
| Release Flow | Large | Scheduled | Medium | Large organizations |


## Key Principles for Any Strategy

Regardless of the branching strategy chosen:

1. **Keep branches short-lived:** Long-lived branches accumulate debt and create merge conflicts.

2. **Protect `main`:** Require PR reviews, CI passing, and approvals before merging to `main`.

3. **Automate testing:** A robust test suite is essential for any branching strategy.

4. **Keep `main` deployable:** The default branch should always be in a state that can be deployed.

5. **Communicate:** The team must agree on and follow the same strategy.

6. **Clean up:** Delete merged branches regularly to keep the repository organized.

The "best" branching strategy is the one your team actually follows consistently. Start simple (GitHub Flow) and add complexity (Gitflow) only when genuinely needed.
