# Trunk-Based Development

Trunk-Based Development (TBD) is a source-control branching model where all developers commit to a single shared branch called the "trunk" (e.g., `main`). Branches are either avoided entirely or kept extremely short-lived (hours, not days).

## Core Principle

```
main (trunk): ─────●─────●─────●─────●───── (everyone commits here)
```

All developers work on the trunk directly or on branches that are merged within hours/days.

## Why Trunk-Based Development?

**Problems with long-lived branches:**
- Merge conflicts accumulate over time
- Integration surprises at merge time
- Slow feedback loop

**Trunk-based development solves these by:**
- Forcing integration constantly (no surprises)
- Running CI on every commit
- Making every developer aware of what others are doing

Google, Facebook, and other high-performing engineering organizations use trunk-based development.

## Two Models of TBD

### 1. Direct Commits to Trunk (True TBD)

Small teams with high trust:

```bash
# Everyone commits directly to main
git pull origin main
# ... make small change...
git commit -m "feat: add email validation"
git push origin main
```

### 2. Short-Lived Feature Branches

Popular for teams using GitHub/GitLab with code review requirements:

```bash
# Create a branch
git switch -c feature/email-validation

# Work for at most 1-2 days
git commit -m "Add email validation"

# Push and create PR
git push -u origin feature/email-validation
# PR reviewed within hours, merged to main

# Delete branch immediately after merge
git branch -d feature/email-validation
```

## Feature Flags Are Essential

Large features that can't be completed in a short branch need **feature flags**:

```javascript
// config/features.js
const features = {
  newCheckoutFlow: process.env.FEATURE_NEW_CHECKOUT === 'true',
  darkMode: process.env.FEATURE_DARK_MODE === 'true',
};

// In your code
if (features.newCheckoutFlow) {
  return newCheckoutHandler(req, res);
}
return legacyCheckoutHandler(req, res);
```

This allows incomplete features to be merged to trunk without being visible to users.

## Requirements for Successful TBD

1. **Comprehensive automated tests** — Every commit runs the full test suite.
2. **Fast CI pipeline** — Feedback within 5-10 minutes.
3. **Feature flags** — For incomplete features.
4. **Small commits** — Developers commit often (multiple times per day).
5. **Strong code review culture** — Quick reviews so branches don't live long.
6. **Discipline** — The team must commit to keeping main always green.

## Branch Naming and Lifetime

In TBD, branches that DO exist should be:

```bash
# Very short-lived (hours to 1-2 days)
git switch -c fix/auth-null-check    # Lives < 1 day
git switch -c feat/email-validation  # Lives < 1-2 days
```

Long-lived branches are anti-patterns in TBD.

## Release Management in TBD

For versioned releases, create **release branches** from trunk at release time:

```bash
# When ready to release v2.0:
git switch main
git switch -c release/v2.0

# Only bug fixes go onto the release branch (cherry-picked from trunk)
git cherry-pick abc1234  # Bug fix from trunk

# Tag the release
git tag -a v2.0.0 -m "Release v2.0.0"
```

## Comparison: TBD vs GitHub Flow vs Gitflow

| Aspect | Trunk-Based | GitHub Flow | Gitflow |
|--------|------------|-------------|---------|
| Branch lifetime | Hours | Days | Weeks |
| PRs | Optional | Required | Required |
| Integration frequency | Continuous | Per PR | Per sprint |
| Suitable for | High-trust teams | Most teams | Scheduled releases |
| CI/CD dependency | Critical | Important | Optional |
