# Monorepo with Git

A **monorepo** is a single Git repository that contains multiple projects, services, or packages. Used by Google, Facebook, Microsoft, and many startups, monorepos offer unique advantages—and challenges—compared to having separate repositories for each project.

## What is a Monorepo?

```
monorepo/
  packages/
    api/              ← Backend API service
    web/              ← Frontend web app
    mobile/           ← Mobile app
    shared-utils/     ← Shared utility library
    design-system/    ← Shared UI components
  tools/
    scripts/
  .github/
    workflows/
  package.json        ← Workspace root
```

## Monorepo vs Polyrepo

| Aspect | Monorepo | Polyrepo |
|--------|---------|---------|
| Code sharing | Easy (shared imports) | Requires publishing packages |
| Atomic changes | Yes (one commit, multiple packages) | Requires coordinated PRs |
| Discovery | All code visible | Fragmented across repos |
| Clone size | Large | Small per repo |
| CI/CD | Needs smart filtering | Simpler per repo |
| Permissions | Shared across all | Per-repo access control |

## Git Features for Monorepos

### Sparse Checkout

Only check out the parts of the repo you work on:

```bash
git clone --no-checkout https://github.com/company/monorepo.git
cd monorepo
git sparse-checkout init --cone

# Only work on the API package
git sparse-checkout set packages/api shared-utils

git checkout main
# Working tree only contains packages/api/ and shared-utils/
```

### Shallow Clone

```bash
# For CI: only clone recent history
git clone --depth=1 https://github.com/company/monorepo.git
```

### Path-scoped git log

```bash
# See changes to only one package
git log --oneline -- packages/api/

# What changed in the design system?
git log -- packages/design-system/
```

## Monorepo Tools

### Nx

```bash
npx create-nx-workspace@latest my-workspace
cd my-workspace

# Run tests only for affected packages
npx nx affected --target=test --base=main

# Build all packages
npx nx run-many --target=build --all
```

### Turborepo

```bash
npx create-turbo@latest
cd my-monorepo

# Run build across all packages
npx turbo run build

# Only rebuild what changed
npx turbo run build --filter=[HEAD^1]
```

### Lerna

```bash
npx lerna init
# Manages npm package versioning and publishing in a monorepo
npx lerna version
npx lerna publish
```

## Path-Based CI/CD

Only run CI for packages that actually changed:

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:

jobs:
  detect-changes:
    runs-on: ubuntu-latest
    outputs:
      api-changed: ${{ steps.changes.outputs.api }}
      web-changed: ${{ steps.changes.outputs.web }}
    steps:
      - uses: actions/checkout@v4
      - uses: dorny/paths-filter@v3
        id: changes
        with:
          filters: |
            api:
              - 'packages/api/**'
            web:
              - 'packages/web/**'

  test-api:
    needs: detect-changes
    if: needs.detect-changes.outputs.api-changed == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: cd packages/api && npm test

  test-web:
    needs: detect-changes
    if: needs.detect-changes.outputs.web-changed == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: cd packages/web && npm test
```

## Best Practices for Monorepos

1. **Use workspace tools** (npm workspaces, Yarn workspaces, pnpm workspaces)
2. **Implement path-based CI/CD** — Don't run all tests on every change
3. **Enforce ownership** with CODEOWNERS
4. **Use sparse checkout** for large repos
5. **Consider Nx or Turborepo** for intelligent caching and task orchestration
6. **Define clear package boundaries** — packages should be as independent as possible
7. **Version packages consistently** — Use Changesets or Lerna for versioning
