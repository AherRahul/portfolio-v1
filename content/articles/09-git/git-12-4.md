# Release Branch Strategy

Release branches provide a stable snapshot that can be tested, hardened, and deployed independently from ongoing feature development. This pattern is essential for products with scheduled versioned releases.

## When to Use Release Branches

Release branches are appropriate when:
- You have **scheduled release cycles** (monthly, quarterly)
- Your product has **multiple simultaneous versions** (v1.x and v2.x both supported)
- You need **extended testing** before releasing to production
- You need a **code freeze** period before release

## The Release Branch Lifecycle

```
develop: ──●──●──●──●──●──●──●──●── (ongoing feature work)
                   │               
release/v1.2: ─────●──●──● ── tagged v1.2.0
                        (only bugfixes)
                   
main: ─────────────────────────●── (merged from release branch)
```

## Creating a Release Branch

```bash
# Create release branch from develop (or main in GitHub Flow)
git switch develop
git pull origin develop
git switch -c release/v1.2.0

git push -u origin release/v1.2.0
```

At this point, `develop` continues with features for v1.3, while `release/v1.2.0` is frozen.

## Release Branch Rules

Only these types of commits are allowed on a release branch:
- 🐛 Bug fixes
- 📝 Documentation updates
- 🔧 Configuration changes (deployment config, env files)
- 🔒 Security patches

**No new features** should be added to a release branch.

## Release Candidate (RC) Tagging

During the testing phase, create RC tags:

```bash
# First release candidate
git tag -a v1.2.0-rc.1 -m "Release candidate 1 for v1.2.0"
git push origin v1.2.0-rc.1

# After fixing beta bugs, RC2
git tag -a v1.2.0-rc.2 -m "Release candidate 2 for v1.2.0"
git push origin v1.2.0-rc.2
```

CI can automatically deploy RC tags to a staging environment for testing.

## Final Release Tag

When the release is approved:

```bash
# Create the final release tag
git switch release/v1.2.0
git tag -a v1.2.0 -m "Release v1.2.0

Features:
- User dashboard
- Export to CSV
- Email notification system

Bug fixes:
- Fix session timeout on mobile
- Fix export button visibility
"

git push origin v1.2.0

# Create GitHub release
gh release create v1.2.0 \
  --title "v1.2.0 — User Dashboard" \
  --notes-from-tag
```

## Merging Back to Main and Develop

After release, merge changes back:

```bash
# Merge to main (for deployable state)
git switch main
git merge release/v1.2.0
git push origin main

# Merge back to develop (bring in bug fixes made on release branch)
git switch develop
git merge release/v1.2.0
git push origin develop
```

This ensures bug fixes made during the release process aren't lost in future development.

## Hotfix on a Release Branch

If a critical bug is found after release:

```bash
# Create hotfix branch from the release tag
git switch -c hotfix/v1.2.1 v1.2.0

# Make the fix
git commit -m "fix: critical security patch"

# Tag the hotfix
git tag -a v1.2.1 -m "Hotfix release v1.2.1: Security patch"
git push origin hotfix/v1.2.1
git push origin v1.2.1

# Merge back to main AND develop
git switch main && git merge hotfix/v1.2.1
git switch develop && git merge hotfix/v1.2.1

# Clean up
git branch -d hotfix/v1.2.1
```

## Automated Release Branch Management

```yaml
# .github/workflows/release.yml
name: Release

on:
  push:
    branches: ['release/v*']
    tags: ['v*.*.0', 'v*.*.*-rc*']

jobs:
  test-and-deploy-staging:
    if: contains(github.ref, 'release/') || contains(github.ref, '-rc')
    environment: staging
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm test
      - run: ./deploy.sh staging

  deploy-production:
    if: "!contains(github.ref, '-rc') && !contains(github.ref, '-alpha')"
    needs: [deploy-staging-final]
    environment: production
    runs-on: ubuntu-latest
    steps:
      - run: ./deploy.sh production
      
  create-release:
    if: startsWith(github.ref, 'refs/tags/v') && !contains(github.ref, '-')
    runs-on: ubuntu-latest
    steps:
      - uses: softprops/action-gh-release@v2
        with:
          generate_release_notes: true
```
