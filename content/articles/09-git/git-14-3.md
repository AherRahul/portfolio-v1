# Git References and HEAD

Git references are human-readable names for SHA-1 hashes. They make navigating commit history much easier than remembering 40-character hash strings. Understanding references is key to mastering Git's power.

## What is a Reference?

A **reference** (or "ref") is a pointer to a specific commit. The three main types of references are:

1. **Branches** — Movable pointers (advance with commits)
2. **Tags** — Fixed pointers (don't move after creation)
3. **HEAD** — A special pointer to the currently checked-out commit

References are stored in `.git/refs/`:

```
.git/
  refs/
    heads/        ← Local branches
      main        ← Points to latest commit on main
      feature/x   ← Points to latest commit on feature/x
    remotes/      ← Remote tracking branches
      origin/
        main
    tags/         ← Tags
      v1.0.0
  HEAD            ← Points to current branch (or commit)
```

## HEAD

`HEAD` is the most important reference — it tells Git where you are right now:

```bash
# See what HEAD points to
cat .git/HEAD
# ref: refs/heads/main   ← Normal branch mode

# When detached:
cat .git/HEAD
# abc1234def5678...      ← Direct commit hash
```

HEAD usually points to a **branch** (not a commit directly). When you commit, that branch moves forward, and HEAD moves with it.

## Inspecting References

```bash
# What commit does HEAD point to?
git rev-parse HEAD
# abc1234...

# What commit does main point to?
git rev-parse main
# def5678...

# What commit does origin/main point to?
git rev-parse origin/main

# List all references
git show-ref
```

## Relative References

Instead of SHA-1 hashes, you can navigate the commit graph relatively:

### Tilde Notation (~): First Parent

```bash
HEAD     # Current commit
HEAD~1   # 1 commit back (same as HEAD^)
HEAD~2   # 2 commits back
HEAD~5   # 5 commits back

# Equivalent for a branch
main~3   # 3 commits back from main's tip

# With a specific commit hash
abc1234~2  # 2 commits before abc1234
```

### Caret Notation (^): Specific Parent

For merge commits (which have multiple parents), `^` selects which parent:

```bash
HEAD^    # First parent (default, same as HEAD~1)
HEAD^2   # Second parent (for merge commits)
HEAD^1   # Same as HEAD^

# Example: merge commit M has parents A and B
M^       # = A (first parent, usually the branch that was merged into)
M^2      # = B (second parent, the incoming branch)
```

## Symbolic References

Git uses symbolic references to create aliases:

```bash
# HEAD is a symbolic ref — it points to a branch, not directly to a commit
git symbolic-ref HEAD
# refs/heads/main

# Set HEAD explicitly (advanced)
git symbolic-ref HEAD refs/heads/feature/login
```

## The MERGE_HEAD, ORIG_HEAD, FETCH_HEAD

Git uses special refs during specific operations:

```bash
ORIG_HEAD   # Previous value of HEAD (saved before merge/rebase/reset)
MERGE_HEAD  # The commit being merged in (during a merge)
FETCH_HEAD  # The result of the last git fetch
CHERRY_PICK_HEAD  # The commit being applied during cherry-pick
```

Useful for recovery:

```bash
# Undo the last merge
git reset --hard ORIG_HEAD

# Undo the last rebase
git reset --hard ORIG_HEAD
```

## The Refspec

A **refspec** maps local references to remote references:

```bash
# Standard push refspec
origin/main:main   # "push local main to origin/main"

# Fetch refspec (in .git/config after clone)
+refs/heads/*:refs/remotes/origin/*
```

You'll see this in `.git/config`:

```ini
[remote "origin"]
    url = https://github.com/you/project.git
    fetch = +refs/heads/*:refs/remotes/origin/*
```
