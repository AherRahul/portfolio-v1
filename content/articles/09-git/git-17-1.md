# Git Submodules

Git submodules allow you to include one Git repository within another as a subdirectory. This is useful when you want to include external projects (like libraries or shared code) in your own repository while keeping them tracked as separate repositories.


## What are Submodules?

A submodule is a way to include one Git repository as a subdirectory of another. The parent repository (the "superproject") stores:
- The submodule's repository URL
- The specific commit the submodule should be at

The submodule's content is NOT stored in the parent repository — only a reference to it.


## Adding a Submodule

```bash
# Add a repository as a submodule
git submodule add <url> [path]

# Examples:
git submodule add https://github.com/org/shared-library.git
git submodule add https://github.com/org/theme.git src/themes/custom
```

This creates a `.gitmodules` file:

```ini
[submodule "shared-library"]
    path = shared-library
    url = https://github.com/org/shared-library.git

[submodule "src/themes/custom"]
    path = src/themes/custom
    url = https://github.com/org/theme.git
```

Commit the `.gitmodules` file and the submodule directory reference:

```bash
git add .gitmodules shared-library
git commit -m "Add shared-library as submodule"
```


## Cloning a Repository with Submodules

When you clone a project with submodules, the submodule directories are empty by default:

```bash
# Clone with all submodules initialized and updated
git clone --recurse-submodules https://github.com/you/project.git

# Or after a regular clone:
git clone https://github.com/you/project.git
cd project
git submodule init    # Register submodule paths
git submodule update  # Download submodule content

# Shorthand combined:
git submodule update --init --recursive
```


## Updating Submodules

When the upstream submodule repository has new commits you want to include:

```bash
# Update a specific submodule to its latest remote commit
cd shared-library
git pull origin main
cd ..

# Commit the updated reference in the parent repo
git add shared-library
git commit -m "Update shared-library to latest version"

# Or update all submodules at once
git submodule update --remote --merge
```


## Working with Submodule Changes

If you need to make changes TO the submodule:

```bash
cd shared-library
git switch -c my-fix
# Make changes
git commit -m "Fix authentication issue"
git push origin my-fix
cd ..

# Update the parent repo to point to your changes
git add shared-library
git commit -m "Point to fixed version of shared-library"
```


## Removing a Submodule

Removing a submodule requires a few steps:

```bash
# 1. Deinitialize
git submodule deinit -f shared-library

# 2. Remove from .git/modules
rm -rf .git/modules/shared-library

# 3. Remove the directory
git rm -f shared-library

# 4. Commit
git commit -m "Remove shared-library submodule"
```


## Submodule Status

```bash
# Check submodule status
git submodule status
# - abc1234 shared-library (v1.2.0)  ← '-' means not initialized
# + def5678 shared-library (v1.2.0)  ← '+' means newer than recorded commit
#   g9h0i1j shared-library (v1.2.0)  ← ' ' means up to date

# List all submodules
git submodule foreach 'echo $path'
```


## When to Use Submodules (and When Not To)

**Good use cases for submodules:**
- Including a library your team maintains separately
- Embedding a third-party framework when you need precise version control
- Sharing code between multiple repositories where the shared code is large

**Consider alternatives:**
- **npm/pip/gem packages:** Use package managers instead of submodules for third-party libraries in most cases.
- **git subtree:** Simpler than submodules for one-way inclusion without tracking external commits.
- **Monorepos:** Keep everything in one repository to avoid submodule complexity.

Submodules add complexity to the workflow — contributors must be aware of them and use `--recurse-submodules` when cloning. Only use them when the benefits clearly outweigh this added complexity.
