# Git's Directed Acyclic Graph (DAG)

At its core, Git's commit history is a **Directed Acyclic Graph (DAG)**. Every commit points to its parent(s), forming a graph that can never have cycles. Understanding this graph model is the key to mastering Git.

## What is a DAG?

**Directed**: Edges (connections) have direction — a commit points TO its parent, not the other way.

**Acyclic**: No cycles — you can never follow parent links and end up at the same commit twice.

**Graph**: Multiple nodes (commits) connected by edges (parent relationships).

## A Simple Commit History as a Graph

```
A ← B ← C ← D  (main)
```

- Each commit points to its **parent** (leftward arrows)
- D is the most recent commit (HEAD)
- A is the initial commit (no parent)

## Branching Creates Multiple Paths

```
A ← B ← C ← D  (main)
         ↑
         └── E ← F  (feature)
```

- C is the **common ancestor** (branch point)
- Both D and F exist independently without knowing about each other

## Merging Creates Multiple Parents

```
A ← B ← C ← D ← M  (main)
         ↑       ↑
         └── E ← F  (merged)
```

- M is a **merge commit** — it has TWO parents: D and F
- This is the ONLY way a commit can have multiple parents in Git
- The graph is still acyclic

## Viewing the Graph

```bash
# ASCII graph in terminal
git log --graph --oneline --all

# Example output:
# * def5678 (HEAD -> main) Merge feature branch
# |\
# | * abc1234 (feature) Add dark mode
# | * 789abcd Update styles
# * | 456cdef Fix button bug
# |/
# * 123cdef Common ancestor
```

## Key Graph Concepts

### Reachability

A commit X is **reachable** from commit Y if you can follow parent links from Y to X:

```bash
# Is abc1234 an ancestor of HEAD?
git merge-base --is-ancestor abc1234 HEAD
echo $?  # 0 = yes, 1 = no

# Find the common ancestor of two commits
git merge-base main feature/dashboard
```

### Branch Tips

Branches are just **named pointers** to specific commits (tip nodes in the graph):

```bash
# What commit does 'main' point to?
git rev-parse main

# What commit does HEAD point to?
git rev-parse HEAD

# Show all branch pointers
cat .git/refs/heads/main
cat .git/refs/heads/feature/dashboard
```

### Divergence

Two branches have **diverged** when neither is an ancestor of the other:

```
main:    A ← B ← C ← D
feature: A ← B ← C ← E ← F
```

Here `main` and `feature` have diverged after C. Neither D nor F is an ancestor of the other.

```bash
# Count commits each branch is ahead/behind
git rev-list --count main..feature  # Commits on feature not in main
git rev-list --count feature..main  # Commits on main not in feature
```

## Commit Graph File (.git/objects/info/commit-graph)

For large repositories, Git builds a commit-graph file to speed up graph traversal:

```bash
# Build the commit graph file
git commit-graph write --reachable

# Or automatically during fetch
git config fetch.writeCommitGraph true
```

Operations like `git log`, `git merge-base`, and `git rebase` are dramatically faster with the commit graph file.

## How Rebase Modifies the Graph

```
Before rebase:
main:    A ← B ← C ← D
feature: A ← B ← E ← F

After git rebase main (from feature):
main:    A ← B ← C ← D
feature:             A ← B ← C ← D ← E' ← F'
```

Rebase creates **new commits** (E', F') with the same changes but different parents. The old E and F still exist until garbage collected.

## Topological Ordering

Git can list commits in topological order (parents always come before children):

```bash
git log --topo-order --oneline --all

# Useful when commits depend on each other in complex ways
# (e.g., after merges and rebases)
```
