# Git Security Best Practices

Git repositories can expose sensitive information if not managed carefully. From accidentally committed secrets to unverified commits, proper security practices protect both your code and your credentials.

## Never Commit Secrets

The #1 Git security rule: **secrets must never enter your repository**, even briefly.

### Prevention Tools

```bash
# Install git-secrets (AWS)
git secrets --install
git secrets --register-aws   # Scans for AWS credentials

# Install pre-commit (Python)
pip install pre-commit
```

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/Yelp/detect-secrets
    rev: v1.4.0
    hooks:
      - id: detect-secrets
        args: ['--baseline', '.secrets.baseline']
  
  - repo: https://github.com/gitleaks/gitleaks
    rev: v8.18.0
    hooks:
      - id: gitleaks
```

```bash
# Install hooks
pre-commit install

# Scan entire history for secrets
gitleaks detect --source . --verbose
trufflehog git file://$(pwd)  # Alternative
```

## Remove Exposed Secrets

If a secret was committed:

```bash
# 1. Immediately rotate the exposed credential!
# 2. Remove from history
git filter-repo --path .env --invert-paths --force
git push --force --all
git push --force --tags

# 3. Notify your security team
# 4. GitHub secret scanning may alert you and block it
```

## GPG Commit Signing

Sign commits to prove authenticity — shows "Verified" badge on GitHub:

```bash
# Generate GPG key
gpg --full-generate-key
# Choose: RSA and RSA, 4096 bits, never expires

# Get key ID
gpg --list-secret-keys --keyid-format=long
# Look for: sec   rsa4096/KEY_ID_HERE

# Configure git to use it
git config --global user.signingkey KEY_ID_HERE
git config --global commit.gpgsign true
git config --global tag.gpgsign true

# Export public key for GitHub
gpg --armor --export KEY_ID_HERE
# Add to GitHub: Settings → SSH and GPG keys → New GPG key
```

Now all your commits and tags are signed. Define branch protective rules to require signed commits.

## SSH Key Security

```bash
# Generate a strong key
ssh-keygen -t ed25519 -C "your@email.com"

# Use a passphrase!
# Never share your private key (~/.ssh/id_ed25519)

# Review authorized keys regularly
cat ~/.ssh/authorized_keys

# Check what keys GitHub knows about
gh api user/keys

# Revoke old/unused keys on GitHub
# Settings → SSH and GPG keys → Delete
```

## Personal Access Token (PAT) Best Practices

```bash
# Use fine-grained PATs (GitHub) with minimum permission
# Never use classic PATs with broad scope
# Set an expiry date (never "no expiration")
# Use one PAT per purpose
# Store in a secret manager, not in shell history

# Rotate PATs regularly
# Review at GitHub: Settings → Developer settings → Personal access tokens
```

## GitHub Security Settings

Organization-level controls:
- **2FA enforcement** — Require all members to enable 2FA
- **IP allowlist** — Restrict to your company's IP ranges
- **SSO (SAML)** — Single Sign-On with your identity provider
- **Audit log** — Track all actions (who pushed, who changed settings)

Repository-level controls:
- **Branch protection rules** — Prevent force pushes to main
- **Secret scanning** — GitHub scans for known secret patterns
- **Dependabot** — Alerts for vulnerable dependencies
- **Code scanning** — Static analysis (CodeQL) for vulnerabilities

## Auditing Access

```bash
# GitHub CLI: list collaborators
gh api repos/{owner}/{repo}/collaborators

# List deploy keys
gh api repos/{owner}/{repo}/keys

# View organization members
gh api orgs/{org}/members

# Full audit log (organization paid plans)
gh api orgs/{org}/audit-log
```

## Environment Secrets in CI/CD

```yaml
# ❌ NEVER do this
env:
  API_KEY: "sk-live-1234567890"  # Hardcoded!

# ✅ ALWAYS use secrets
env:
  API_KEY: ${{ secrets.PROD_API_KEY }}  # From GitHub Secrets
```

Manage secrets:
- **GitHub:** Settings → Secrets and variables → Actions
- **Environment secrets:** Settings → Environments → environment secrets (requires approval)
