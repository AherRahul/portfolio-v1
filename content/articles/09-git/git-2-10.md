# git commit — Saving Snapshots

`git commit` creates a permanent snapshot of the staged changes in your repository's history. Every commit records *who* made the change, *when*, and *why* (through the commit message).

## What Happens When You Commit

1. Git takes everything in the staging area.
2. Creates a **commit object** with your metadata (author, date, message).
3. Links it to the previous commit as its parent.
4. Advances the branch pointer (e.g., `main`) to this new commit.

## Basic Usage

```bash
# Opens the default editor to write a commit message
git commit

# Commit with an inline message
git commit -m "Add user authentication"

# Stage all modified tracked files and commit in one step
git commit -am "Fix bug in login validation"
# Note: -am doesn't stage NEW untracked files — only modified tracked files
```

## Writing Good Commit Messages

The standard format (adapted from Tim Pope's convention):

```
Short summary (50 chars max)

Optional longer body explaining WHY the change was made,
what problem it solves, and any caveats. Wrap at 72 chars.

Footer:
Closes #234
Co-authored-by: Jane Doe <jane@example.com>
```

**Rules for the subject line:**
- Capitalize the first word
- No period at the end
- Use imperative mood: "Add feature" not "Added feature"
- Keep under 50 characters
- Summarize WHAT was done, not HOW

## Commit Options

```bash
# Amend the most recent commit (message and/or files)
git commit --amend

# Amend without changing the message
git commit --amend --no-edit

# Commit with a specific author
git commit --author="Jane Doe <jane@example.com>" -m "message"

# Commit with a specific date
git commit --date="2024-01-01T10:00:00" -m "Past commit"

# Dry run — show what would be committed
git commit --dry-run

# Commit without running hooks
git commit --no-verify -m "Skip pre-commit hook"
```

## Understanding Commit Objects

Every commit stores:
- A reference to the root **tree** (the directory snapshot)
- Reference to **parent commit(s)**
- **Author** name, email, timestamp
- **Committer** name, email, timestamp
- The **commit message**

```bash
# View the raw commit object
git cat-file -p HEAD
```

## Atomic Commits

An **atomic commit** is one that:
- Represents a single logical change
- Can be understood, reviewed, reverted independently
- Passes all tests on its own

```bash
# Good: atomic commits
git add src/auth.js
git commit -m "Add JWT token generation"

git add tests/auth.test.js
git commit -m "Add JWT token unit tests"

# Avoid: mega-commits
git add .
git commit -m "Add auth, fix bug, update docs, and refactor DB"
```

## Best Practices

1. **Commit early and often** on local branches — you can always squash later.
2. **Never commit broken code** to shared branches.
3. **Don't commit secrets** — use `.gitignore` and environment variables.
4. **Match tense in team** — agree on imperative ("Add") vs past tense ("Added").
