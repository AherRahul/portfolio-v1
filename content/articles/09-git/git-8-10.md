# Splitting Commits

Sometimes a commit contains too many unrelated changes. Splitting a commit into multiple smaller, focused commits makes your history clearer and easier to review. This is done using interactive rebase.


## Why Split Commits?

A commit like "Add authentication and fix payment bug and update docs" should really be three separate commits:
1. "Add user authentication"
2. "Fix null pointer in payment processing"
3. "Update authentication documentation"

Splitting makes each commit:
- **Easier to review** in pull requests
- **Easier to revert** if something goes wrong
- **More useful for `git bisect`** when debugging

## How to Split a Commit

### Step 1: Start Interactive Rebase

```bash
git rebase -i HEAD~3   # Or however far back the commit is
```

### Step 2: Mark the Commit for Editing

Change `pick` to `edit` for the commit you want to split:

```
pick abc1234 Minor fixes
edit def5678 Big commit with too many things   ← Mark this
pick g9h0i1j Latest feature
```

Save and close.

### Step 3: Git Pauses at the Commit

Git applies the commit and pauses:

```
Stopped at def5678...  Big commit with too many things
You can amend the commit now...
```

### Step 4: Reset the Commit

Reset the commit while keeping the changes in your working directory (unstage everything from this commit):

```bash
git reset HEAD~1
```

Now all the changes from the "big commit" are unstaged in your working directory.

### Step 5: Create Multiple Focused Commits

Stage and commit the changes in logical groups:

```bash
# First: commit authentication changes
git add src/auth.js src/token.js tests/auth.test.js
git commit -m "Add user authentication with JWT"

# Second: commit the payment bug fix
git add src/payment.js
git commit -m "Fix null pointer exception in payment processing"

# Third: commit documentation updates
git add docs/auth.md README.md
git commit -m "Update authentication documentation"
```

### Step 6: Continue the Rebase

```bash
git rebase --continue
```

Git will replay the remaining commits (in our example, `g9h0i1j`).


## Using `git add -p` for Precise Splitting

When a single file has multiple unrelated changes, use `git add -p` to interactively select which hunks to stage:

```bash
git reset HEAD~1   # Unstage the big commit

git add -p src/complex-file.js   # Interactively select hunks
# Git shows each change and asks: y (yes), n (no), s (split), e (edit)

git commit -m "First focused change"

git add -p src/complex-file.js   # Select remaining hunks
git commit -m "Second focused change"

git rebase --continue
```


## After Splitting

Since splitting rewrites history, force-push is needed for already-pushed branches:

```bash
git push --force-with-lease origin feature/your-branch
```

## Summary

| Step | Command |
|------|---------|
| Start interactive rebase | `git rebase -i HEAD~N` |
| Mark commit for editing | Change `pick` to `edit` |
| Undo the commit (keep changes) | `git reset HEAD~1` |
| Create first focused commit | `git add <files>; git commit -m "..."` |
| Create second focused commit | `git add <files>; git commit -m "..."` |
| Continue the rebase | `git rebase --continue` |
