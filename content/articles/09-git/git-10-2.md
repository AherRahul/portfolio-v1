# Forking Workflow

The forking workflow is the standard approach for contributing to open source projects and for team setups where not everyone has direct write access to the main repository. Instead of pushing directly to the central repository, contributors fork (copy) the repository, make their changes, and submit a pull request.


## What is a Fork?

A **fork** is a copy of a repository that lives in your own GitHub account. It's completely independent of the original (called the "upstream") but maintains a connection that allows you to sync changes and submit pull requests.

```
Upstream Repo (original-owner/project)
       ↓ Fork
Your Fork (your-username/project)  ←→  Local Clone
           ↓ Pull Request
Upstream Repo ← Code contribution
```


## Setting Up the Forking Workflow

### Step 1: Fork the Repository

On GitHub, click the **"Fork"** button on the repository page. This creates `your-username/project` in your account.

### Step 2: Clone Your Fork

```bash
git clone https://github.com/your-username/project.git
cd project
```

### Step 3: Add the Upstream Remote

```bash
git remote add upstream https://github.com/original-owner/project.git

# Verify your remotes
git remote -v
# origin    https://github.com/your-username/project.git (fetch)
# origin    https://github.com/your-username/project.git (push)
# upstream  https://github.com/original-owner/project.git (fetch)
# upstream  https://github.com/original-owner/project.git (push)
```


## Contributing: The Workflow

### Step 1: Sync with Upstream Before Starting

Always start from the latest upstream state:

```bash
git fetch upstream
git switch main
git merge upstream/main    # or: git rebase upstream/main
git push origin main       # Keep your fork's main up to date
```

### Step 2: Create a Feature Branch

Never work directly on `main`. Create a branch for each contribution:

```bash
git switch -c feature/improve-documentation
```

### Step 3: Make Your Changes and Commit

```bash
# Edit files...
git add .
git commit -m "docs: improve installation instructions"
```

### Step 4: Push to Your Fork

```bash
git push -u origin feature/improve-documentation
```

### Step 5: Create a Pull Request

On GitHub, go to your fork and click **"Create pull request"**. This opens a PR from your fork's branch to the original repository's `main` branch.


## Keeping Your Fork Synchronized

Over time, the upstream repository will receive new commits. Keep your fork in sync:

```bash
# Option 1: Fetch and merge
git fetch upstream
git switch main
git merge upstream/main
git push origin main

# Option 2: Fetch and rebase (cleaner)
git fetch upstream
git switch main
git rebase upstream/main
git push origin main --force-with-lease  # Only needed if you rebased
```


## After Your PR is Merged

```bash
# Delete the feature branch locally
git branch -d feature/improve-documentation

# Delete from your fork (remote)
git push origin --delete feature/improve-documentation

# Sync your fork with upstream
git fetch upstream
git switch main
git merge upstream/main
git push origin main
```


## Benefits of the Forking Workflow

1. **No write access needed** to the original repository — perfect for open source.
2. **Clean separation** — contributors work in their own isolated copy.
3. **Maintainers control** what gets merged — the PR is the gating mechanism.
4. **Easy experimentation** — your fork is your playground without affecting others.


## Tips for Contributing to Open Source

1. **Read CONTRIBUTING.md** before starting — understand the project's contribution guidelines.
2. **Start small** — your first PR to a project should be focused and small.
3. **Test your changes** before submitting.
4. **Follow the code style** of the project.
5. **Write clear PR descriptions** — maintainers review many PRs.
6. **Be patient** — maintainers are often volunteers.
