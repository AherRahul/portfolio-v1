# Docker and Git Workflow

Docker and Git work closely together in modern deployment pipelines. Git provides version-controlled source code, and Docker packages it into consistent, deployable images.

## The Git-Docker Connection

```
Git commit → CI/CD trigger → Docker build → Image push → Deploy
```

Every Docker image should be traceable to a specific Git commit.

## Build Docker Images with Git Metadata

```dockerfile
# Dockerfile
FROM node:20-alpine AS base
WORKDIR /app

# Build arguments for Git metadata
ARG GIT_COMMIT=unknown
ARG GIT_BRANCH=unknown
ARG BUILD_DATE=unknown

# Embed metadata as labels
LABEL org.opencontainers.image.revision=$GIT_COMMIT
LABEL org.opencontainers.image.created=$BUILD_DATE
LABEL git.branch=$GIT_BRANCH

COPY package*.json ./
RUN npm ci --only=production

COPY . .

CMD ["node", "src/server.js"]
```

Build with Git metadata:

```bash
docker build \
  --build-arg GIT_COMMIT=$(git rev-parse HEAD) \
  --build-arg GIT_BRANCH=$(git branch --show-current) \
  --build-arg BUILD_DATE=$(date -u +%Y-%m-%dT%H:%M:%SZ) \
  -t myapp:$(git rev-parse --short HEAD) .
```

## Docker Image Tagging Strategy

Tag Docker images the same way you tag Git branches and releases:

```bash
# Get Git info
GIT_HASH=$(git rev-parse --short HEAD)
GIT_TAG=$(git describe --tags --exact-match 2>/dev/null || echo "")
BRANCH=$(git branch --show-current)

# Tag with commit hash (always)
docker build -t myapp:$GIT_HASH .

# Tag with branch name (for feature branches)
docker build -t myapp:$BRANCH .

# Tag as latest if on main
if [ "$BRANCH" = "main" ]; then
  docker tag myapp:$GIT_HASH myapp:latest
fi

# Tag with semantic version if on a release tag
if [ -n "$GIT_TAG" ]; then
  docker tag myapp:$GIT_HASH myapp:$GIT_TAG
  # Also tag as major version
  MAJOR=$(echo $GIT_TAG | cut -d. -f1)
  docker tag myapp:$GIT_HASH myapp:$MAJOR
fi
```

## GitHub Actions: Build and Push Docker Image

```yaml
# .github/workflows/docker.yml
name: Build Docker Image

on:
  push:
    branches: [main]
    tags: ['v*']

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Docker metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ghcr.io/your-org/your-app
          tags: |
            type=ref,event=branch
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}
            type=sha,prefix=sha-

      - name: Login to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

## Multi-Stage Builds with Development/Production

```dockerfile
# Dockerfile
# Stage 1: Builder
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: Production
FROM node:20-alpine AS production
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY --from=builder /app/dist ./dist
CMD ["node", "dist/server.js"]

# Stage 3: Test (for CI)
FROM builder AS test
RUN npm test
```

Different targets:
```bash
# Production build
docker build --target production -t myapp:prod .

# Run tests in Docker
docker build --target test -t myapp:test .
```

## Rollback with Docker Tags

The Git-Docker combination makes rollback trivial:

```bash
# Services are deployed with specific Git-tagged image versions
# Rolling back = deploying the previous image tag

# List available images (corresponds to Git tags)
docker images myapp --format "{{.Tag}}\t{{.CreatedAt}}"
# sha-abc1234    2024-01-15 (current, broken)
# sha-def5678    2024-01-14 (previous, good)
# v1.2.0         2024-01-10 (stable release)

# Rollback to previous stable release
kubectl set image deployment/myapp myapp=myapp:v1.2.0
```
