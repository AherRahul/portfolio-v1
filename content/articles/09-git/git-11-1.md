# Gitflow Workflow

Gitflow is a branching model invented by Vincent Driessen in 2010. It provides a robust framework for managing releases and is popular in projects with scheduled release cycles.


## The Core Branches

Gitflow has two long-lived branches:

1. **`main`** — Contains production-ready code. Each commit on `main` is a release.
2. **`develop`** — The integration branch where features are merged. Contains the latest development changes.

```
main:    ───●─────────────────●───── (releases)
             \               /
develop: ─────●───●───●───●─── (continuous development)
```


## Supporting Branches

Gitflow uses several types of short-lived supporting branches:

### Feature Branches

- **Branch from:** `develop`
- **Merge back to:** `develop`
- **Naming:** `feature/feature-name`

```bash
# Create feature branch
git switch develop
git switch -c feature/user-authentication

# ... work on feature ...

# Merge back to develop
git switch develop
git merge --no-ff feature/user-authentication
git branch -d feature/user-authentication
```

### Release Branches

- **Branch from:** `develop`
- **Merge back to:** `develop` **and** `main`
- **Naming:** `release/v1.2.0`

Release branches are for final testing, bug fixing, and release preparation. No new features go in.

```bash
# Create release branch when develop is ready for release
git switch develop
git switch -c release/v1.2.0

# Only bug fixes and release preparations from here
git commit -m "Bump version to 1.2.0"

# When ready: merge to main and tag
git switch main
git merge --no-ff release/v1.2.0
git tag -a v1.2.0 -m "Release v1.2.0"

# Also merge back to develop
git switch develop
git merge --no-ff release/v1.2.0

git branch -d release/v1.2.0
```

### Hotfix Branches

- **Branch from:** `main`
- **Merge back to:** `develop` **and** `main`
- **Naming:** `hotfix/fix-description`

For critical production bugs that can't wait for the next release.

```bash
# Branch from main (production code)
git switch main
git switch -c hotfix/critical-xss-vulnerability

# Fix the bug
git commit -m "Fix XSS vulnerability in user input"

# Merge to main and tag
git switch main
git merge --no-ff hotfix/critical-xss-vulnerability
git tag -a v1.2.1 -m "Hotfix v1.2.1"

# Also merge to develop!
git switch develop
git merge --no-ff hotfix/critical-xss-vulnerability

git branch -d hotfix/critical-xss-vulnerability
```

```
               │     │              │

               │                   │

               │                  v1.1.0
               │
hotfix: ───────└── hotfix/bug ──┐
                                └── (merged to main + develop)
```


## When to Use Gitflow

✅ **Good for:**
- Projects with scheduled, version-based releases
- Teams that support multiple versions simultaneously
- Projects requiring a formal release process with staging/QA

❌ **Less ideal for:**
- Continuous deployment projects (where you push to production daily)
- Small teams with simple workflow needs
- Projects using GitHub Flow or trunk-based development

## Gitflow Tools

The `git-flow` command-line extension simplifies the workflow:

```bash
# Install git-flow
brew install git-flow    # macOS

# Initialize gitflow in a repository
git flow init

# Create a feature branch
git flow feature start user-authentication

# Finish and merge the feature
git flow feature finish user-authentication

# Create and finish releases
git flow release start 1.2.0
git flow release finish 1.2.0

# Create and finish hotfixes
git flow hotfix start critical-bug
git flow hotfix finish critical-bug
```
