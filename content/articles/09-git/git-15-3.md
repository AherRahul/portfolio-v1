# git worktree — Multiple Working Trees

`git worktree` allows you to have multiple working trees (working directories) attached to a single Git repository. This means you can check out multiple branches simultaneously in different directories without the overhead of separate clones.

## The Problem

Traditional workflow when you need to work on a hotfix while mid-feature:

```bash
# Option 1: Stash and switch (disruptive)
git stash
git switch main
git switch -c hotfix/urgent
# ... fix bug ...
git switch feature/dashboard
git stash pop

# Option 2: Create another clone (wastes disk space, has separate .git/)
git clone /path/to/repo /path/to/repo-hotfix
```

`git worktree` provides a better solution: check out another branch in a new directory, sharing the same `.git/` database.

## Basic Usage

```bash
# Create a new worktree for a branch
git worktree add ../project-hotfix hotfix/urgent-fix

# Or create a new branch in the worktree
git worktree add -b hotfix/urgent-fix ../project-hotfix main
```

This creates:
```
/path/to/project/           ← Main worktree (feature branch)
/path/to/project-hotfix/    ← New worktree (hotfix branch)
```

Both directories share the same `.git/` repository and object database.

## Working in Multiple Worktrees

```bash
# Terminal 1: Continue feature work
cd /path/to/project
git status   # On branch feature/dashboard

# Terminal 2: Work on the hotfix
cd /path/to/project-hotfix
git status   # On branch hotfix/urgent-fix
# ... make fix, commit, push ...
```

## Listing Worktrees

```bash
git worktree list
```

Output:
```
/path/to/project            abc1234 [feature/dashboard]
/path/to/project-hotfix     def5678 [hotfix/urgent-fix]
```

## Removing a Worktree

```bash
# Delete the worktree directory first (optional)
rm -rf /path/to/project-hotfix

# Then prune the reference
git worktree prune

# Or remove and prune in one step
git worktree remove project-hotfix
```

## Constraints

- You **cannot check out the same branch in two worktrees** simultaneously.
- Each worktree has its own staging area and HEAD.
- The `.git/` directory in additional worktrees is a file pointing to the main `.git/` directory.

```bash
# Attempting to checkout a branch already checked out elsewhere fails
git worktree add ../another-hotfix hotfix/urgent-fix
# fatal: 'hotfix/urgent-fix' is already checked out at '...'
```

## Practical Use Cases

```bash
# 1. Review a PR while working on your own feature
git worktree add ../review-pr-456 origin/feature/pr-456-branch
cd ../review-pr-456
# Review and test the PR
cd ../project
git worktree remove ../review-pr-456

# 2. Keep a "main" always-compiled worktree for reference
git worktree add ../project-main main

# 3. Build different versions simultaneously
git worktree add ../build-v1 v1.x
git worktree add ../build-v2 v2.x
cd ../build-v1 && npm run build
cd ../build-v2 && npm run build
```

## Benefits Over Multiple Clones

| Aspect | Worktree | Multiple Clones |
|--------|---------|-----------------|
| Disk usage | Shared objects | Duplicated objects |
| Refs shared | Yes | No |
| Remote config | Shared | Separate |
| Object database | Shared | Separate |
