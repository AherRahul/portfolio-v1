# Dependency Management with Git

Dependencies are a critical part of any project. Git has specific best practices for how to track, update, and secure your dependencies within version control.

## What to Commit for Dependencies

### Lockfiles — Always Commit Them

```
package-lock.json      ← npm lockfile
yarn.lock              ← Yarn lockfile
pnpm-lock.yaml         ← pnpm lockfile
Gemfile.lock           ← Ruby Bundler
poetry.lock            ← Python Poetry
Pipfile.lock           ← Pipenv
composer.lock          ← PHP Composer
go.sum                 ← Go modules
Cargo.lock             ← Rust Cargo
```

**Why:** Lockfiles ensure that every developer and CI environment installs the exact same dependency versions. Without lockfiles, `npm install` might install different versions on different machines.

### node_modules — NEVER Commit Them

```gitignore
# .gitignore
node_modules/
.venv/
vendor/      # (if not using vendor mode intentionally)
__pycache__/
*.pyc
```

These directories are generated from your lockfiles and should NOT be in version control.

## Committing Lockfile Updates

When your dependencies change:

```bash
# Install a new dependency
npm install express

# This updates package.json AND package-lock.json
# Commit BOTH together
git add package.json package-lock.json
git commit -m "chore(deps): add express v4.18"

# Updating all dependencies
npm update
git add package-lock.json
git commit -m "chore(deps): update all dependencies"
```

## Automating Dependency Updates

### Dependabot (GitHub Native)

```yaml
# .github/dependabot.yml
version: 2
updates:
  # npm dependencies
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
    open-pull-requests-limit: 10
    groups:
      # Group all dev dependencies into one PR
      dev-dependencies:
        dependency-type: "development"
    ignore:
      # Don't update major versions automatically
      - dependency-name: "*"
        update-types: ["version-update:semver-major"]

  # GitHub Actions
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "monthly"
```

Dependabot creates PRs automatically when new versions are available. Your CI verifies they don't break anything.

### Renovate (More Configurable)

```json
// renovate.json
{
  "extends": ["config:base"],
  "packageRules": [
    {
      "matchDepTypes": ["devDependencies"],
      "automerge": true,
      "automergeType": "pr",
      "matchUpdateTypes": ["patch", "minor"]
    },
    {
      "matchPackageNames": ["react", "vue", "next"],
      "groupName": "framework",
      "stabilityDays": 7
    }
  ],
  "schedule": ["before 9am on Monday"]
}
```

## Reviewing Dependency PRs

When reviewing a Dependabot/Renovate PR:

1. **Check the version diff** — What changed from v2.1 to v2.2?
2. **Read the changelog** — Click the release notes link in the PR
3. **Check for breaking changes** — Look for BREAKING CHANGE or major version changes
4. **Verify CI passes** — Tests should catch most regression
5. **Review `.diff` of lockfile** — Long but confirms no unexpected transitive dependency changes

## Vendoring Dependencies

Some projects commit dependencies directly ("vendoring"):

```bash
# Go modules
go mod vendor      # Copies dependencies to vendor/ folder
git add vendor/
git commit -m "chore: vendor all Go dependencies"

# PHP Composer
composer install --no-dev
git add vendor/
```

**Pros:** Fully reproducible without external registries, works offline  
**Cons:** Repository size increases significantly, PRs are large when updating

## Security Auditing

```bash
# Find known vulnerabilities
npm audit
pnpm audit
yarn audit

# Auto-fix vulnerabilities
npm audit fix

# View audit as JSON for CI integration
npm audit --json > audit-report.json

# GitHub Actions: automatically audit dependencies
```

```yaml
# .github/workflows/security.yml
jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm audit --audit-level=high
```
