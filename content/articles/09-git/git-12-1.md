# git tag

Tags in Git are references that point to specific commits in history. They're most commonly used to mark **release points** — capturing the state of the code at a particular version (e.g., `v1.0.0`, `v2.3.1`).


## Types of Tags

### Lightweight Tags

A lightweight tag is just a pointer to a specific commit — essentially a named bookmark. It stores no extra information.

```bash
git tag v1.0.0
```

### Annotated Tags

An annotated tag is stored as a full object in Git's database. It includes:
- Tagger name and email
- Date
- A message
- Can be signed with GPG

```bash
git tag -a v1.0.0 -m "Release version 1.0.0"
```

**Recommendation:** Use annotated tags for releases — they contain valuable metadata.


## Creating Tags

```bash
# Create a lightweight tag on current commit
git tag v1.0.0

# Create an annotated tag
git tag -a v1.0.0 -m "Release version 1.0.0"

# Tag a specific past commit
git tag -a v0.9.0 abc1234 -m "Version 0.9.0 beta"

# Signed tag (requires GPG key)
git tag -s v1.0.0 -m "Signed release v1.0.0"
```


## Listing Tags

```bash
# List all tags
git tag

# List tags matching a pattern
git tag -l "v1.*"   # All v1.x tags

# Show tag details (annotated tags)
git show v1.0.0
```


## Pushing Tags to Remote

Tags are NOT pushed automatically with `git push`. You must push them explicitly:

```bash
# Push a single tag
git push origin v1.0.0

# Push all local tags not yet on remote
git push --tags origin

# Push with commits and tags together
git push origin main --tags
```


## Deleting Tags

```bash
# Delete local tag
git tag -d v1.0.0-beta

# Delete remote tag
git push origin --delete v1.0.0-beta
# or
git push origin :refs/tags/v1.0.0-beta
```


## Checking Out a Tag

To view the repository state at a tag (enters detached HEAD state):

```bash
git checkout v1.0.0

# Or to create a branch from a tag
git checkout -b hotfix/v1.0.1 v1.0.0
```


## Semantic Versioning

Most projects use **Semantic Versioning (SemVer)** for version numbers:

```
v<MAJOR>.<MINOR>.<PATCH>
```

- **MAJOR:** Breaking changes (not backward compatible)
- **MINOR:** New features (backward compatible)
- **PATCH:** Bug fixes (backward compatible)

Examples:
- `v1.0.0` → `v1.0.1` (patch: bug fix)
- `v1.0.0` → `v1.1.0` (minor: new feature)
- `v1.0.0` → `v2.0.0` (major: breaking change)

Pre-release tags: `v2.0.0-alpha.1`, `v2.0.0-beta.3`, `v2.0.0-rc.1`


## Tag Workflow for Releases

```bash
# 1. Make sure main is clean and up to date
git switch main
git pull origin main

# 2. Create annotated release tag
git tag -a v1.2.0 -m "Release v1.2.0: Add payment processing

Features:
- Stripe payment integration
- Invoice generation
- Payment history view

Bug fixes:
- Fix null pointer in checkout (closes #456)
"

# 3. Push the tag
git push origin v1.2.0

# 4. Create GitHub Release (on GitHub.com) from the tag
```


## Summary

```bash
git tag                         # List all tags
git tag v1.0.0                  # Create lightweight tag
git tag -a v1.0.0 -m "msg"     # Create annotated tag
git tag -a v1.0.0 <hash>       # Tag a specific commit
git show v1.0.0                 # Show tag details
git push origin v1.0.0          # Push one tag
git push --tags origin          # Push all tags
git tag -d v1.0.0               # Delete local tag
git push origin --delete v1.0.0 # Delete remote tag
git checkout v1.0.0             # Checkout tag (detached HEAD)
```
