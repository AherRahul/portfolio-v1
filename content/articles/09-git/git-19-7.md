# Kubernetes and GitOps

GitOps extends Git workflow to infrastructure — your cluster's desired state is defined in a Git repository, and an operator automatically reconciles the actual state to match.

## What is GitOps?

GitOps principles (from Weaveworks):

1. **Declarative** — The system state is described declaratively (YAML configs in Git)
2. **Versioned and immutable** — Git is the single source of truth
3. **Pulled automatically** — Software agents (ArgoCD, Flux) pull state from Git
4. **Continuously reconciled** — Agents ensure cluster state matches Git state

```
Git Repository → GitOps Operator → Kubernetes Cluster
(desired state)    (ArgoCD/Flux)    (actual state)
```

## Repository Structure

Common patterns for organizing GitOps repositories:

### Monorepo Pattern

```
gitops-repo/
  apps/
    staging/
      frontend/
        deployment.yaml
        service.yaml
        ingress.yaml
      backend/
        deployment.yaml
    production/
      frontend/
      backend/
  infrastructure/
    cert-manager/
    ingress-nginx/
    prometheus/
  base/
    frontend/   ← Kustomize or Helm base configs
    backend/
```

### App + Infra Repos

```
app-repo/          ← Application code (developers work here)
  src/
  Dockerfile
  .github/workflows/build-push.yml  ← Builds image, updates gitops-repo

gitops-repo/       ← Deployment configs (DevOps/Platform team)
  apps/
  infrastructure/
```

## ArgoCD: GitOps for Kubernetes

ArgoCD watches your Git repository and syncs the cluster:

```yaml
# argocd/apps/frontend-staging.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: frontend-staging
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/org/gitops-repo.git
    targetRevision: HEAD
    path: apps/staging/frontend
  destination:
    server: https://kubernetes.default.svc
    namespace: staging
  syncPolicy:
    automated:
      prune: true       # Remove deleted resources
      selfHeal: true    # Revert manual cluster changes
```

## The GitOps Deployment Flow

```bash
# 1. Developer commits application code
git commit -m "feat: add dark mode toggle"
git push origin main

# 2. CI builds Docker image with Git SHA as tag
# GitHub Actions:
docker build -t myapp:abc1234 .
docker push ghcr.io/org/myapp:abc1234

# 3. CI updates the GitOps repo (auto-PR or direct commit)
# In gitops-repo:
sed -i 's|image: ghcr.io/org/myapp:.*|image: ghcr.io/org/myapp:abc1234|' \
  apps/staging/frontend/deployment.yaml
git commit -am "chore: deploy frontend abc1234 to staging"
git push

# 4. ArgoCD detects the change and applies it
# Cluster now runs the new image

# 5. After validation, promote to production
sed -i 's|image: ghcr.io/org/myapp:.*|image: ghcr.io/org/myapp:abc1234|' \
  apps/production/frontend/deployment.yaml
git commit -am "chore: deploy frontend abc1234 to production"
git push
```

## Kustomize for Environment Overlays

```yaml
# base/kustomization.yaml
resources:
  - deployment.yaml
  - service.yaml

# overlays/staging/kustomization.yaml
bases:
  - ../../base
patchesStrategicMerge:
  - deployment-patch.yaml

# overlays/staging/deployment-patch.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: frontend
spec:
  replicas: 1   # Staging uses 1 replica
  template:
    spec:
      containers:
        - name: frontend
          image: ghcr.io/org/myapp:abc1234  ← Updated by CI
```

## Rollback in GitOps

```bash
# Option 1: Git revert (recommended - maintains audit trail)
git revert abc1234   # The commit that changed the image tag
git push
# ArgoCD detects and rolls back the cluster automatically

# Option 2: Git reset (dev environments only)
git reset --hard HEAD~1
git push --force-with-lease
```

The power of GitOps: rollback is just a Git operation with full audit trail.
