# git bisect

`git bisect` is a powerful debugging tool that uses **binary search** to find the specific commit that introduced a bug. Instead of manually checking each commit, Git automatically narrows down the culprit in O(log n) steps.


## The Problem git bisect Solves

Imagine you discover a bug in your application today. You know it wasn't there 3 months ago, and there have been 500 commits since then. Manually checking each commit would take forever. `git bisect` can identify the offending commit in just ~9 steps (log₂(500) ≈ 9).


## How Binary Search Works in Git Bisect

1. You tell Git: "This commit is bad" and "This commit was good."
2. Git checks out the middle commit between them.
3. You test and tell Git "good" or "bad."
4. Git halves the range and repeats.
5. After ~log₂(N) steps, Git identifies the first bad commit.


## Basic Workflow

```bash
# Start the bisect session
git bisect start

# Mark the current commit as bad (the bug is present here)
git bisect bad
# or specify a commit: git bisect bad <hash>

# Mark a known good commit (before the bug existed)
git bisect good v1.0.0
# or: git bisect good abc1234
```

Git now checks out the middle commit:

```
Bisecting: 250 revisions left to test after this (roughly 8 steps)
[def5678...] Commit message
```

**Test the application at this commit.** Then:

```bash
# If the bug is present at this commit:
git bisect bad

# If the bug is NOT present at this commit:
git bisect good
```

**Repeat** until Git identifies the culprit:

```
abc1234... is the first bad commit
commit abc1234...
Author: Developer <dev@example.com>
Date:   Mon Jan 1 2024

    Refactor authentication logic
```

### End the Session

```bash
# Return to the original HEAD
git bisect reset
```


## Automated Bisect with a Test Script

If you have a test script that exits with 0 (pass) or non-zero (fail), bisect can run fully automatically:

```bash
git bisect start
git bisect bad HEAD          # Current state is broken
git bisect good v2.0.0       # Known good version

# Run an automated test script
git bisect run npm test      # npm test returns 0 for pass, non-zero for fail

# Or a custom script:
git bisect run ./scripts/test-feature.sh
```

Git will automatically go through all commits and report the first bad one, all without manual input.


## Skipping Commits

Sometimes a commit can't be tested (e.g., build is broken at that commit):

```bash
git bisect skip
# or
git bisect skip <commit-hash>

# Skip a range of commits
git bisect skip abc1234..def5678
```

Git will avoid marking skipped commits as bad or good and find a nearby commit to test instead.


## Real-World Example

```bash
# Bug report: "The login form broke sometime in the last 2 weeks"
git bisect start
git bisect bad HEAD

# Two weeks ago was roughly the v3.1.0 release
git bisect good v3.1.0

# Git checks out middle commit — test your login form
# ... test shows bug is present ...
git bisect bad

# Git checks out earlier middle commit
# ... test shows bug is NOT present ...
git bisect good

# Continue until Git says: "abc1234 is the first bad commit"
git bisect reset

# Now you know exactly which commit to investigate!
git show abc1234
git diff abc1234^ abc1234
```


## Tips for Effective Bisecting

1. **Write a reproducible test case first** — the clearer the test, the faster the bisect.
2. **Automate when possible** — `git bisect run` is much faster than manual testing.
3. **Keep commits small and focused** — smaller commits make the culprit easier to identify once found.
4. **Document strange behavior** in commit messages so future bisects are easier.
