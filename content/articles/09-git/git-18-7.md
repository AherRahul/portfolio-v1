# Enterprise Git Workflows

Large enterprises with hundreds or thousands of developers face Git challenges that small teams don't: complex release coordination, cross-team dependencies, compliance requirements, and massive repository sizes.

## Hierarchical Repository Ownership

Enterprise Git often uses a hierarchical structure:

```
Company Git Server
├── Platform Team
│   └── platform-core (upstream for all projects)
├── Product Team A
│   ├── service-user
│   ├── service-payments
│   └── shared-components
└── Product Team B
    ├── service-analytics
    └── service-reports
```

Teams own their repositories and use **inner source** (using open-source contribution patterns internally between teams).

## The Inner Source Model

When Team B needs a feature in Team A's code:

```bash
# Team B member forks Team A's repo internally
git clone https://git.company.com/team-a/shared-components.git
# OR creates a branch if they have access:
git switch -c feature/team-b-need

# Makes changes, opens PR to Team A's repo
# Team A reviews and merges
```

This prevents the "Big Ball of Mud" problem where anyone can change anyone's code without review.

## Enterprise Branching Strategies

For enterprise with quarterly releases:

```
main           ─────────────────────────────── (always deployable, auto-deploy to staging)
develop        ──────────●──●──●──●──────────── (integration branch for features)
release/2024Q1 ──────────────────●──●──●─── (frozen, only bug fixes)
hotfix/urgent  ──────────────────────────●── (critical fix, merged to main + release)
```

## Compliance and Audit Requirements

Many enterprises need:

```yaml
# Enforced via GitHub branch protection:
✅ Require PR for every change (no direct push to main/develop)
✅ Require 2+ approvals (4-eyes principle)
✅ Require specific team approval (CODEOWNERS)
✅ Require signed commits (GPG)
✅ Require linear history (no merge commits)
✅ Audit log enabled (all actions tracked)
```

Audit trail configuration:

```bash
# Every commit shows signed verification
git log --show-signature --oneline

# Check who approved which PR
gh api repos/{owner}/{repo}/pulls/{number}/reviews
```

## Dependency Management at Scale

Cross-team dependencies require careful versioning:

```bash
# Internal package registry (e.g., GitHub Packages, Nexus)
npm publish --registry https://npm.company.com

# Consuming team pins versions
"@company/shared-components": "^2.3.0"

# Automated updates via Renovate/Dependabot
# create PRs when @company/shared-components has new version
```

## Repository Size Management

For large enterprises:

```bash
# Virtual File System for Git (Git VFS) - used by Microsoft
# Allows multiple TBs of code without full clone

# Or: Sparse checkout for selective cloning
git sparse-checkout set my-team-package/

# Or: Partial clone
git clone --filter=blob:none https://git.company.com/monorepo.git

# Or: Split into multiple microrepos and use aggregator tooling
```

## Enterprise CI/CD Governance

```yaml
# Centralized workflow templates (GitHub)
# .github/workflows/enterprise-ci.yml (in .github repo)

name: Enterprise CI Template

on:
  workflow_call:
    secrets:
      SONAR_TOKEN:
        required: true

jobs:
  security-scan:
    runs-on: [self-hosted, enterprise]  # Company-managed runners
    steps:
      - uses: actions/checkout@v4
      - name: SAST Scan
        run: ./enterprise-security-scan.sh

  compliance-check:
    runs-on: ubuntu-latest
    steps:
      - name: License compliance
        run: ./check-licenses.sh
```

Teams call the centralized template:

```yaml
# Each team's .github/workflows/ci.yml
jobs:
  enterprise-standards:
    uses: company/.github/.github/workflows/enterprise-ci.yml@main
    secrets:
      SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
  
  team-specific-tests:
    runs-on: ubuntu-latest
    steps:
      - run: npm test
```

## Self-Hosted Runners

Enterprise often requires self-hosted CI runners for security:

```yaml
runs-on: [self-hosted, linux, x64, production-runner]
```

Running on company infrastructure ensures:
- Code never leaves the corporate network
- Access to internal services
- Compliance with data residency requirements
- Custom hardware (GPUs for ML teams, large RAM)

## Incident Response with Git

Enterprise incident playbook:

```bash
# 1. Identify the bad commit
git bisect start
git bisect bad HEAD
git bisect good v2024.01.15  # Last known good release

# 2. Document the incident
git commit -m "incident: INC-2024-01-16 - auth service outage

Root cause: abc1234 removed null check causing NPE
Impact: 1 hour auth downtime, ~5000 users affected
Resolution: revert abc1234

Refs: INC-2024-01-16"

# 3. Communicate via commit message context
# 4. Post-mortem → action items → issues → PRs
```
