# GitHub Fork

A **fork** is a personal copy of another user's repository. Forking is the foundation of open source software development on GitHub, allowing anyone to contribute to any public project without needing direct write access.


## What Does Forking Do?

When you fork a repository on GitHub:
1. GitHub creates a **complete copy** of the repository in your account.
2. The fork is **independent** — your changes don't affect the original.
3. The fork **maintains a link** to the original (called the "upstream" repository).
4. You have **full write access** to your fork.

```
Original Repo → Fork in Your Account → Local Clone
(read-only for you)   (you have full access)
```


## Fork vs. Clone

| Aspect | Fork | Clone |
|--------|------|-------|
| Creates copy on | GitHub (server-side) | Your local computer |
| Access permissions | Full write to your fork | Depends on original access |
| Relationship to original | Linked as "upstream" | Remote is the original |
| Use case | Open source contributions | Working on a project you have access to |

**Cloning** downloads a repository to your computer. **Forking** creates a server-side copy on GitHub.

Typically you **fork first, then clone your fork**.


## How to Fork on GitHub

1. Navigate to the repository you want to fork.
2. Click the **"Fork"** button (top right).
3. Optionally change the name and description.
4. Click **"Create fork"**.

GitHub creates `https://github.com/your-username/repository-name`.


## After Forking: Clone Your Fork

```bash
# Clone your fork (not the original)
git clone https://github.com/your-username/repository-name.git
cd repository-name

# Add the original as "upstream"
git remote add upstream https://github.com/original-owner/repository-name.git

# Verify remotes
git remote -v
# origin    https://github.com/your-username/repository-name.git (fetch)
# origin    https://github.com/your-username/repository-name.git (push)
# upstream  https://github.com/original-owner/repository-name.git (fetch)
# upstream  https://github.com/original-owner/repository-name.git (push)
```


## Keeping Your Fork Updated

As the original project evolves, sync your fork:

```bash
# Fetch latest upstream changes
git fetch upstream

# Merge into your local main
git switch main
git merge upstream/main

# Update your fork on GitHub
git push origin main
```


## Contributing Back via Pull Request

After making changes on a feature branch in your fork:

```bash
# Push your feature branch to your fork
git push -u origin feature/my-contribution

# On GitHub: open a PR from your fork's branch to the original repo's main
# Original repo → Pull requests → New pull request → Compare across forks
```


## The Full Open Source Contribution Cycle

```
1. FORK the original repository on GitHub

2. CLONE your fork locally:
   git clone https://github.com/you/project.git

3. ADD upstream remote:
   git remote add upstream https://github.com/original/project.git

4. SYNC before starting work:
   git fetch upstream && git merge upstream/main

5. CREATE feature branch:
   git switch -c fix/typo-in-readme

6. MAKE changes and COMMIT:
   git add README.md && git commit -m "docs: fix typo in installation section"

7. PUSH to your fork:
   git push -u origin fix/typo-in-readme

8. CREATE PULL REQUEST on GitHub

9. RESPOND to code review feedback

10. PR MERGED by maintainer ✓

11. CLEAN UP: delete branch, sync fork
```

Forking is what makes open source collaboration at scale possible — millions of developers can independently experiment and contribute without interfering with each other.
