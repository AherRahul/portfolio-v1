# git cherry-pick — Real-World Use Cases

While the `git cherry-pick` command itself is straightforward, knowing *when* and *why* to use it separates effective Git users from beginners. Here are the most common, practical scenarios.

## Use Case 1: Applying a Hotfix to Multiple Branches

You have a critical bug fix on `main` that also needs to be in `release/v1.x` and `release/v2.x`:

```bash
# Get the commit hash of the fix on main
git log --oneline main | head -5
# abc1234 Critical: fix XSS vulnerability in input sanitizer

# Apply to release/v1.x
git switch release/v1.x
git cherry-pick abc1234
git push origin release/v1.x

# Apply to release/v2.x
git switch release/v2.x
git cherry-pick abc1234
git push origin release/v2.x
```

Use `-x` to automatically add a reference to the original commit:

```bash
git cherry-pick -x abc1234
# Commit message becomes:
# "Critical: fix XSS vulnerability in input sanitizer
# (cherry picked from commit abc1234)"
```

## Use Case 2: Moving a Commit to the Right Branch

You committed to `main` by accident, but it should be on a feature branch:

```bash
# Note the accidental commit hash
git log --oneline main
# abc1234 Add experimental dark mode (should be on feature branch!)
# def5678 Merge PR #45: Update header

# Create the feature branch
git switch -c feature/dark-mode

# Cherry-pick the commit onto the feature branch
git cherry-pick abc1234

# Remove the commit from main (undo it)
git switch main
git revert abc1234   # Or: git reset --hard HEAD~1 if not pushed

# Verify
git switch feature/dark-mode
git log --oneline -3  # Should show the commit
```

## Use Case 3: Salvaging Commits from an Abandoned Branch

A feature branch was abandoned but has some good work worth keeping:

```bash
# See what's on the abandoned branch
git log --oneline abandoned/payment-v2

# Cherry-pick only the useful commits
git switch main
git cherry-pick g9h0i1j   # The useful payment logic
git cherry-pick k2l3m4n   # The related test

# Leave behind the work-in-progress commits
```

## Use Case 4: Backporting Features

A feature built for v3 is needed in the v2 maintenance branch:

```bash
# Feature is on main, needs to go to release/v2.x
git log --oneline main  
# p9q0r1s Add PDF export feature

git switch release/v2.x  

# Cherry-pick (may need to resolve conflicts due to different code base)
git cherry-pick p9q0r1s

# If conflicts arise:
# Resolve conflicts → git add → git cherry-pick --continue
```

## Use Case 5: Selective PR Integration

You have a PR with 5 commits but only want commits 2 and 4:

```bash
# Get the commit hashes from the PR branch
git log --oneline feature/big-pr
# a1b2c3d Commit 5: unrelated change
# d4e5f6g Commit 4: useful utility function
# h7i8j9k Commit 3: work in progress  
# l0m1n2o Commit 2: good bug fix
# p3q4r5s Commit 1: initial draft

# Cherry-pick only commits 2 and 4 (in order)
git switch main
git cherry-pick l0m1n2o   # Commit 2
git cherry-pick d4e5f6g   # Commit 4
```

## Cherry-Pick Range

```bash
# Apply a range of commits from feature branch (commits a..e)
# Does NOT include the first commit
git cherry-pick commita..commite

# Include BOTH boundary commits (use ^ on first)
git cherry-pick commita^..commite
```

## When NOT to Cherry-Pick

❌ **Avoid cherry-pick when:**
- The commits have complex inter-dependencies
- You're cherry-picking large ranges (use rebase/merge instead)
- The same fix needs to go everywhere — better to merge to common ancestor

✅ **Use cherry-pick when:**
- Applying isolated, independent commits
- Backporting critical fixes to older release branches
- Rescuing good work from a messy or abandoned branch
