# Cherry-Pick Conflict Resolution

Cherry-picking commits from diverged branches often leads to conflicts. Understanding how to resolve them efficiently and cleanly is an essential skill.

## Why Cherry-Pick Conflicts Happen

Conflicts occur when the surrounding context of the cherry-picked code differs from the target branch:

```
Branch A:         Branch B (cherry-pick target):
function auth() { function authenticate() {    ← renamed!
  validateUser();   validateCredentials();      ← line changed!
  logIn();          logIn();
}                 }
```

If you cherry-pick a commit that modifies `auth()` from Branch A to Branch B, Git can't find the same context — conflict.

## The Conflict Resolution Workflow

```bash
# Attempt cherry-pick
git cherry-pick abc1234

# If conflicts occur:
# error: could not apply abc1234... Add rate limiting to auth
# hint: After resolving the conflicts, mark them with
# hint: "git add/rm <pathspec>", then run
# hint: "git cherry-pick --continue"

# 1. See which files conflict
git status
# both modified: src/auth.js

# 2. Open and resolve conflicted files
# Look for conflict markers:
# <<<<<<< HEAD
# function authenticate() {
# =======
# function auth() {
# >>>>>>> abc1234 (Add rate limiting)

# 3. Edit to your desired result
# Keep the target branch's naming but apply the change:
# function authenticate() {
#   if (rateLimitExceeded(user)) throw new Error('Rate limit');  ← added
#   validateCredentials();
#   logIn();
# }

# 4. Stage the resolved file
git add src/auth.js

# 5. Continue the cherry-pick
git cherry-pick --continue
# Opens editor for commit message (keep or modify)
```

## Abort vs Continue vs Skip

```bash
# Abort: cancel the cherry-pick entirely, return to previous state
git cherry-pick --abort

# Continue: after resolving conflicts, continue applying the commit
git cherry-pick --continue

# Skip: skip this commit (leave it out) and continue with next
git cherry-pick --skip
```

## Multi-Commit Cherry-Pick Conflicts

When cherry-picking a range:

```bash
git cherry-pick v1.0.0..v1.1.0

# If first commit conflicts:
# Resolve → git add → git cherry-pick --continue
# If second commit conflicts:
# Resolve → git add → git cherry-pick --continue
# Skip a commit: git cherry-pick --skip
```

## Using a Merge Tool

```bash
# Launch your configured merge tool for visual resolution
git mergetool

# Common merge tools:
# VS Code: git config --global merge.tool vscode
# vimdiff: git config --global merge.tool vimdiff
# IntelliJ: git config --global merge.tool intellij
```

After resolving all conflicts in the tool:

```bash
git cherry-pick --continue
```

## Checking the Result

After cherry-pick succeeds:

```bash
# Verify the commit looks correct
git show HEAD

# Compare with the original commit
git show abc1234       # Original
git show HEAD          # Cherry-picked result

# If the cherry-pick added -x flag, verify the reference
git log --oneline -3
# def5678 Fix rate limiting in auth (cherry picked from commit abc1234)
```

## Minimizing Conflicts

1. **Cherry-pick in chronological order** — Dependencies resolve naturally
2. **Cherry-pick close commits first** — Less divergence = fewer conflicts
3. **Rebase before cherry-picking** — Update your branch to recent main first
4. **Keep cherry-picks small** — Smaller commits are easier to resolve
5. **Use `git log --cherry-pick`** — Find commits not already included:
   ```bash
   git log main..feature/my-branch --cherry-pick --no-merges
   ```

## When Cherry-Pick is Too Messy

If conflicts are extensive, consider:
- Doing a full merge instead
- Manually applying the changes
- Rebasing the branch instead of cherry-picking

```bash
# Alternative: generate a patch and apply it manually
git diff abc1234^ abc1234 > my-change.patch
git apply my-change.patch  # Apply to current working tree
git add .
git commit -m "Apply change from abc1234"
```
